\section{Introduction}

This chapter presents a large-scale study of GPU MCS testing that tests $10\times$ more devices than previous studies. The techniques that enable this study are also applied in two other ways: (1)~a deep investigation of the MCS implemented by NVIDIA's A100, H100, GH200, and GB10 GPUs (\mysec\ref{sec:gpuharbor-nvidia-mcs}), and (2)~an investigation into how observations of weak behaviors can be used to create a low-bandwidth GPU side-channel that also provided the foundation for the creation of a more sophisticated side-channel across NVIDIA MIG instances on an H100 GPU (\mysec\ref{sec:gpuharbor-sidechannel}).

\paragraph{GPUHarbor.} \mytablong\ref{tab:gpuharbor-gpu-overview} summarizes the large-scale study, including the number of GPUs tested (\totalDevices), broken down by two frameworks (WebGPU and Vulkan) and seven vendors (Intel, Apple, NVIDIA, AMD, Arm, Qualcomm, and Imagination). This scale is enabled by \gpuharbor, a new cross-platform GPU MCS testing tool suite. \gpuharbor includes two frontends, a browser web app (using WebGPU) and an Android app (using Vulkan). Since WebGPU is now better supported on mobile devices, a similar study today could likely reach more mobile GPUs through the web interface. The web app was advertised on campus forums and social media to obtain a significant number of WebGPU results. Fewer Vulkan devices were tested because the Android app required manual installation.

As in the previous chapter, \gpuharbor uses litmus tests to evaluate MCS behaviors. Prior litmus testing approaches have several shortcomings: (1)~they are implemented in vendor-specific languages, e.g., CUDA; (2)~they require expert users to build, configure, and execute tests on each device, e.g., as is the case for OpenCL; or (3)~litmus tests were embedded in vendor-specific stress testing environments and thus would not execute efficiently on other devices. This cumbersome litmus testing workflow made it infeasible to perform a large-scale study. In contrast, \gpuharbor defines litmus tests using a neutral configuration (written in JSON), which it compiles to a portable shading language (WGSL~\cite{webgpu-sl} or SPIR-V~\cite{spir-v}). The resulting litmus testing application then tunes the testing stress automatically. The net result is a fully automated and easy-to-use tool for GPU MCS testing at large. \mytablong\ref{tab:gpuharbor-gpu-overview} shows how many weak memory litmus test iterations were run and how many weak behaviors were observed in the study.

The data set supports the following two investigations: (1)~an examination of MCS conformance test results that finds two new bugs in mobile device GPUs from Arm and NVIDIA, and (2)~a characterization of weak memory behavior profiles, e.g., the rates at which allowed weak behaviors occur and their sensitivity to system stress. Additionally, several analyses are performed on the weak memory profiles. First, per-vendor average profiles are compared; for example, AMD shows an average weak-behavior rate of 1.5\%, while Intel shows only 0.06\%. Different GPUs are then clustered, revealing that, surprisingly, cross-vendor devices often have similar profiles, while devices from the \textit{same} vendor sometimes have vastly different profiles. Finally, the wide range of observed profiles is discussed in terms of its impact on testing strategies and the implementation of synchronization algorithms.

\paragraph{Litmus Testing NVIDIA GPUs.} NVIDIA and its CUDA programming framework remain leaders in GPU computing today. CUDA also provides a formalized MCS for its PTX intermediate representation~\cite{lustig2019ptx}, making it a prime target for investigation of MCS development and testing. NVIDIA is also the first vendor to officially support cache-coherent interconnects between the CPU and GPU in their Arm-based Grace-Hopper Superchip, motivating an understanding of how the GPU MCS interacts with the CPU MCS. Previous work has developed a compound memory consistency model~\cite{goens2023compound}, which gives semantics to programs whose executions span processors with different MCSs. Intuitively, a compound MCS describes what behaviors are possible when, for example, CPU code and GPU code communicate through shared memory while each side follows its own ordering rules. Yet, the MCS behaviors of NVIDIA's recent GPUs remain empirically unvalidated. Using the techniques from MC Mutants and GPUHarbor, this thesis presents comprehensive MCS testing of NVIDIA A100, H100, GH200, and GB10 GPUs. On the GH200, a set of 560 variants of the message passing test is run on the GPU to characterize its behavior under different memory scopes and synchronization levels. Message passing is chosen because it represents the ubiquitous producer-consumer synchronization pattern found in many real-world applications. A total of 1,302 litmus test variants are also run on each GPU to determine whether it provides multi-copy atomicity, a property that affects the design of new MCSs supporting independent compilation of communicating GPU and CPU code.

\paragraph{Developing MCS-Based Side-Channels.} An MCS in some ways crosses the abstraction barrier between hardware and software, surfacing low-level hardware mechanisms at the programming language level. Though this exposure has so far proved necessary for performance, it also creates a vector for attackers to learn about and exploit hardware-level mechanisms. The GPU MCS testing techniques in this thesis are shown to be one such vector, where stress executed by one process can lead to an increase in the rate of weak behavior observations on another process. The feasibility of this side-channel is evaluated on three GPUs, an NVIDIA RTX 4070, an AMD RX 7900 XT, and an Apple M3. The Apple M3 in particular is shown to provide a \textit{parallel execution model}, where programs from multiple processes can execute on the GPU at the same time, allowing a reliable low-capacity side-channel of 16 bits per second to be created. While this side-channel's capacity is extremely low, it provides a basis for future, more sophisticated attacks. For example, the same side-channel mechanism is shown to work on NVIDIA MIG technology, which allowed for a deeper investigation into the partitioning scheme of this technology and the creation of a high-capacity side-channel attack on NVIDIA Hopper-series GPUs when MIG is enabled.

\section{GPUHarbor System Overview \label{sec:gpuharbor-sys-overview}}

Building on the approaches in \mychap\ref{chap:mc-mutants}, this section discusses the testing campaign (\mysec\ref{sec:gpuharbor-litmus-selection}) and the development of MCS testing tools that are easily accessible on a wide range of devices. The overview covers each stage of the tooling, starting with litmus test generation (\mysec\ref{sec:gpuharbor-litmus-test-gen}), moving on to the design of \gpuharbor's web interface and Android app (\mysec\ref{sec:gpuharbor-website-design}), and ending with the data collection process (\mysec\ref{sec:gpuharbor-data-collection}).

\subsection{Litmus Test Selection \label{sec:gpuharbor-litmus-selection}}

The tests used in the study build on the MCS mutation testing strategy used in \mychap\ref{chap:mc-mutants}. The study uses 32 \textit{mutants}, of which 24 are litmus tests with weak behaviors allowed by the WebGPU MCS. The mutants are used to find effective system stress under which to run the conformance tests. The results analysis focuses on characterizing the rates of weak behaviors of the six mutants shown in \myfig\ref{fig:gpuharbor-mcs-litmustests}. Up to symmetry, these tests enumerate the combinations of four loads and stores, with two operations on each of two threads, that form cyclic weak behaviors over two memory locations. Thus, they capture all pairwise memory-reordering patterns in this class. For example, the \textbf{SB} test checks for store-load reorderings, while the \textbf{LB} test checks for load-store reorderings. Additionally, these tests capture synchronization patterns used in common concurrency algorithms like a compare-and-swap spinlock. Because of this, prior work has also focused on these tests and has shown their utility in finding bugs in both applications and MCS implementations~\cite{sorensen2016exposing,kirkham2020foundations}.

Once the mutants are run, the weak behavior profile of a device is used to determine an effective system stress configuration under which to run conformance tests. The study uses the 20 conformance tests from \mychap\ref{chap:mc-mutants}. As a concrete illustration using one mutant and conformance test, the \textbf{MP} test is run under many different system stress configurations to build a weak behavior profile. The configuration most effective at revealing \textbf{MP} weak behaviors is then used to run the closely related \textbf{MP-CO} conformance test. \textbf{MP-CO} is a version of \textbf{MP} in which \texttt{x} and \texttt{y} are the same memory location, following Mutator 2 from \mysec\ref{sec:mc-mutants-mutator-2}. Its weak behavior violates SC-per-location and is disallowed by most major MCSs. This approach was shown to be effective at finding bugs in \mychap\ref{chap:mc-mutants}, and its effectiveness is further demonstrated by the discovery of two new bugs: a violation of \textbf{MP-CO} on Arm devices and a violation of \textbf{MP-CO} on an NVIDIA device (see \mysec\ref{sec:gpuharbor-vulkan-bugs}).

\subsection{Litmus Test Generation \label{sec:gpuharbor-litmus-test-gen}}

This subsection discusses the tooling that generates and runs the testing and characterization campaign. Litmus test behaviors are non-deterministic and sensitive to system stress. Because of this, the shaders that run the litmus tests contain not only the actual litmus test instructions, like those in \myfig\ref{fig:gpuharbor-mcs-litmustests}, but also take in a number of parameters and provide functions that are used to construct system stress.

To provide a standardized interface for defining litmus tests in different GPU languages, a tool called Litmus Generator (\textsc{LitGen}, \circledWhite{1} in \myfig\ref{fig:gpuharbor-sys-overview}) was built. It is similar to previous litmus testing tools~\cite{alglave2011litmus} but is specifically targeted to create GPU programs with system stress, which has been shown to be necessary for testing GPU MCSs~\cite{alglave2015gpu,kirkham2020foundations}. \textsc{LitGen} takes litmus tests that are written in an abstract format, currently JSON, that specify the actions of the test (e.g.\ loads and stores) and the possible behaviors of the test, with a special designation being given to weak behaviors. The tests used in this work were all manually specified, as they are relatively small, but \textsc{LitGen} could be integrated with other tools that use formal models to generate litmus tests, e.g.,~\cite{alglave2010fences,wickerson2017automatically}, which would provide more automation and account for more complicated tests and MCSs. \textsc{LitGen} outputs a \textit{test shader}, which runs the test alongside system stress developed in prior work~\cite{levine2023mcmutants,kirkham2020foundations}, and a \textit{result shader}, which aggregates the observed behaviors of the test.

The result shader is generated separately from the test shader for several reasons:
\begin{enumerate}
    \item Some tests, like \textbf{2+2W} (\myfig\ref{fig:gpuharbor-22w-litmustest}), examine memory locations for weak behaviors after all threads have finished executing the test. To avoid relying on synchronization features, some of which are being tested, the test memory buffer is instead passed into a new result aggregation shader, which executes after the test shader.
    \item \textsc{LitGen} implements parallel testing, described in \mychap\ref{chap:mc-mutants}, which runs thousands of \textit{instances} of each litmus test concurrently. Thus, it is natural to leverage the inherent parallelism of the GPU to also aggregate the many results, which otherwise may be time-consuming to do on the CPU, especially since it requires copying memory from the GPU to the CPU.
\end{enumerate}

Currently, two backends exist for \textsc{LitGen}. The tool outputs WGSL shaders directly, as WGSL is a text-based language. SPIR-V, on the other hand, is a low-level representation similar to LLVM, increasing its flexibility but making code generation more complex. Therefore, for Vulkan backends, \textsc{LitGen} first outputs OpenCL, a compute-focused GPU language that is similar in syntax to C++. Then, it uses Clspv~\cite{clspv} (\circledWhite{2}), a prototype compiler from OpenCL to SPIR-V, to generate the shader used in the Android app.

As WebGPU is primarily browser-based while Vulkan runs on native devices, litmus testing driver programs are currently maintained in two languages. WebGPU exposes a relatively simple JavaScript API, on which the web interface is built (\circledWhite{3}).

Vulkan's native C/C++ API is more complex, so to simplify this process, a Vulkan compute wrapper called \textsc{EasyVk} was built (\circledWhite{4}). This library exposes a simple interface for defining buffers and dispatching shaders to the GPU in a few lines of code. \textsc{EasyVk} is integrated into the Android app.

\subsection{\gpuharbor Design \label{sec:gpuharbor-website-design}}

\begin{figure}
\hfill
\includegraphics[width=\columnwidth]{images/gpuharbor.png}
\caption{A screenshot of \gpuharbor's web interface showing the ``Explore'' page for the \textbf{MP} litmus test.}
\label{fig:gpuharbor-interface}

Previous GPU MCS studies have been limited in reach due to the difficulty in deploying cross-platform GPU applications~\cite{sorensen2016hitchhiker}. One of the reasons for this is the fractured landscape of GPU development. NVIDIA's popular CUDA framework~\cite{cuda} is used for many data science applications but is only supported on NVIDIA GPUs. OpenCL~\cite{opencl} was introduced in 2009 by Apple as a cross-platform standard; however, today Apple no longer supports OpenCL and instead requires developers to use their proprietary Metal API~\cite{apple_metal}. %and the future of OpenCL remains unclear.

Another issue with previous GPU MCS testing approaches has been a reliance on expert users to run testing campaigns. For example, using OpenCL would require users to install the required drivers, build the application under a specific environment, etc. To collect data from the diversity of devices necessary to gain confidence in cross-platform GPU frameworks, tools must minimize the friction in setting up and running tests. The tools introduced here are easily distributed applications with a user-friendly interface on top of new GPU frameworks so that even non-technical users can collect data on their devices and submit results for analysis. To this end, \gpuharbor is introduced as a GPU MCS testing tool with two widely supported and accessible frontends, a web interface and an Android app (\circledWhite{3} and \circledWhite{5} in \myfig\ref{fig:gpuharbor-sys-overview}).

\gpuharbor's web interface and Android app have a common design with two functions: exploring (\circledWhite{6}) and tuning/conforming (\circledWhite{7}). ``Explore'' pages run specific litmus tests, display histograms of results, and provide the ability to adjust various parameters that control system stress. When tuning and conforming, a set of tests is chosen to run with multiple random system stress configurations, searching for configurations that maximize the rate of weak behaviors and uncover bugs in MCS implementations. While this study includes the largest collection of data on mobile GPU MCS behaviors, future work could increase the reach of mobile GPU MCS testing even further.

\paragraph{Exploring\label{sec:gpuharbor-sys-overview-explore}}

\myfiglong\ref{fig:gpuharbor-interface} shows a screenshot of \gpuharbor's web interface explore page for the \textbf{MP} litmus test after the test has been run with relatively high system stress on a MacBook Pro with an integrated Intel Iris GPU. The top of the page (\circledGrey{1}) includes a description of the test and pseudocode showing the test instructions. The right-hand side (\circledGrey{2}) includes an editable list of the parameters that define system stress, along with several presets. When the test is running, the histogram (\circledGrey{3}) updates in real-time with the number of times each behavior is observed. The progress bar gives an estimate of how much time remains, based on the speed of previous iterations.

The green bars correspond to sequential behaviors, where one thread runs entirely before the other. The blue bar corresponds to interleaved behaviors, where actions from each thread are interleaved (e.g.\ leading to the behavior \texttt{r0 == 0 \&\& r1 == 1} in the \textbf{MP} litmus test). The red bar corresponds to weak behaviors; in this run, three \textbf{MP} weak behaviors were observed out of over 13 million test instances, so the histogram shows behaviors using a log scale, as weak behaviors are relatively rare.

%We have found that test exploration is useful in several ways. When designing experiments, we can quickly iterate on different system stress configurations and number of iterations. It has also proved useful as a pedagogical tool for teaching both GPU programming and the weak behaviors allowed by relaxed MCSs, which we expand on in \mysec\ref{sec:gpuharbor-eval-ped}.

\paragraph{Tuning and Conforming\label{sec:gpuharbor-sys-overview-tune-conform}}

Both the web interface and the Android app can be used to tune system stress. When tuning, a set of tests can be selected, with presets available for weak memory tests (e.g.\ those in \myfig\ref{fig:gpuharbor-mcs-litmustests}) and conformance tests, e.g., to test coherence. Testing options like the number of configurations, the maximum number of workgroups, and other parameter overrides can be modified to run different experiments and check specific tests without redeploying any code.

To collect data from volunteer users across a diverse set of devices, the options users have to configure are minimized. This reduces the chances of errors and provides a standardized dataset to analyze. The web interface's tuning page, therefore, includes a tab that exposes no configuration options, but instead shows only a few buttons: one button that starts a combined tuning/conformance run with default parameters and another button that pulls up a submission form, which submits the results along with some (optional) contact information. All results are anonymized; contact details were only collected if users wanted to be informed about the outcome of the study. Before submitting, users agreed that their anonymized results could be aggregated, reported on, and released as part of this study. The deployment was designed to study GPU behavior rather than human subjects, with submitted data consisting of anonymized test results and device metadata; optional contact information was not used in the analysis, and user agreement was required before submission.

\subsection{Data Collection \label{sec:gpuharbor-data-collection}}

To submit test data, the web interface communicates with a backend service that exposes an API for submitting results and inserting them into an SQLite database (\circledWhite{8} in \myfig\ref{fig:gpuharbor-sys-overview}). The data are then analyzed using Python scripts (\circledWhite{9}).

The Android app is not yet available on the app store, nor is it integrated with the SQLite backend, so results are manually copied from the device for analysis. Future work could reduce the friction for submitting mobile app results and thus increase the reach of future studies. Nevertheless, the study of eight devices is the largest known testing campaign of mobile GPU MCS behaviors.

While system stress configurations are generated randomly, the configurations run on different devices must be the same for data analysis purposes. That is, if different GPUs are tested with the same stress configurations, their behavior under the same stress can be compared. This is ensured by integrating a seedable Park-Miller random number generator~\cite{park1988random} into both the web interface and the Android app and using the same seed when running all tuning experiments.

By default, browsers only expose limited information about the user's GPU without turning on vendor-specific development flags due to privacy and security concerns around fingerprinting~\cite{webgpu-security}. To collect as much information as possible, users were given instructions asking them to temporarily enable flags so detailed GPU information could be collected. Of the 98 results collected, 67 included the exact GPU tested. The other 31 results did not specify the exact GPU, but included only the vendor and a string describing the GPU architecture, such as ``intel gen-9'' or ``nvidia ampere''. All Apple devices reported an architecture of ``common-3'', making it impossible to immediately distinguish M1s from M2s. However, \mysec\ref{sec:gpuharbor-webgpu-sim} shows that the data can be used to infer device information, hindering the ability of browsers to hide the specifics of a user's GPU.

\section{Initial Results: Weak Behavior Characterization \label{sec:gpuharbor-wb-char}}

\begin{figure}
\hfill
\includegraphics[width=\columnwidth]{images/webgpu-timing.pdf}
\caption{Histogram showing the time spent testing WebGPU's MCS on each device using \gpuharbor's web interface. Each bin is broken down by vendor.}
\label{fig:gpuharbor-webgpu-timing}
\end{figure}

To collect data from as many sources as possible, the link to \gpuharbor's web interface was disseminated to the general public using campus forums and social media, and the Android app was run on eight physically accessible devices. As shown in \mytab\ref{tab:gpuharbor-gpu-overview}, the study collected data from millions of tests; each test used a randomly generated system stress configuration, with 50 configurations used on the web interface and 150 on the Android app. In each configuration, tests were run millions of times based on a randomly generated number of workgroups and threads per workgroup.

To ensure data integrity, a checksum algorithm verified that the expected number of overall behaviors was observed based on the system stress configuration. The testing duration was also recorded; however, some computers went to sleep in the middle of the tests, suspending the browser's process and leading to extremely long recorded test times. To account for this issue, testing time was recorded on a per-test/configuration basis, and the results were then filtered to exclude any test/configuration durations over one minute. Each individual test runs quickly (e.g., in less than 5 seconds); thus, runs that took over one minute most likely occurred when the computer went to sleep. To approximate the duration of a suspended test, a neighboring test's time was used.

One consideration when collecting data from the wider public is that tests cannot run for hours at a time. Previous work targeted only a few devices, running tests on one device for a minimum of 36 hours~\cite{kirkham2020foundations} or 2 hours (\mychap\ref{chap:mc-mutants}). However, asking volunteer users to leave their browsers and computers open for that long is impractical and would certainly decrease the number of submissions. Therefore, the number of test environments and iterations per environment was chosen heuristically, aiming for the tests to finish in 10--20 minutes.

\myfiglong\ref{fig:gpuharbor-webgpu-timing} shows the distribution of testing time on the web interface, broken down by vendor. The results show that NVIDIA devices were the fastest on average, mostly running all tests in under 15 minutes. On the other hand, Intel devices ran slower, with two older Intel GPUs taking over an hour and a half to complete.

The rest of this section analyzes the WebGPU and Vulkan data to characterize the rates at which weak behaviors occur on devices from different vendors. These initial results motivate three research questions, which are explored in depth in \mysec\ref{sec:gpuharbor-insight-impact}:

\begin{enumerate}
    \item Do MCS bugs exist in the wild, especially in GPUs that are relatively untested (\mysec\ref{sec:gpuharbor-vulkan-bugs})?
    \item Can the characterization data be used to identify similarities between GPUs (\mysec\ref{sec:gpuharbor-webgpu-sim})? If so, then the data can be used to develop new testing strategies or to expose potential new browser fingerprinting vulnerabilities.
    \item How can a weak behavior characterization study be used in programming guides for implementing synchronization constructs, e.g.\ mutexes (\mysec\ref{sec:gpuharbor-vulkan-locks})?
\end{enumerate}

Intel devices tended to have the lowest rate of weak behaviors, with just over half of them (15/26) revealing weak behaviors on each test. The median rate of weak behaviors on Intel devices was even lower than their average, around 0.02\% for each test. No Intel device showed a rate of weak behaviors above 1\% on any test.

Interestingly, that device was also the only device not to observe any weak behaviors on \textbf{S}, \textbf{LB}, and \textbf{2+2W}. The only other device not to reveal weak behaviors on a test was a Quadro K620 with a Maxwell architecture, on \textbf{MP}, \textbf{R}, and \textbf{SB}.

Apple devices were consistently weak, revealing weak behaviors on every device and test, generally at a higher rate on all tests than NVIDIA devices but with less variation than AMD devices. Apple GPUs have only recently been built into non-mobile devices, so these results represent the first comprehensive evaluation of weak behaviors on Apple GPUs. Although the specific name of every Apple device was unavailable, enough information was collected to show that the results include Apple M1 (basic, Pro, Max) and Apple M2 (basic, Pro) devices.

AMD devices were also very weak, with 100\% of devices showing weak behaviors on every test. The clear highest average rate occurs on the \textbf{SB} litmus test on AMD GPUs. Most of the AMD devices show a high rate of weak behaviors on \textbf{SB}, approaching 10\% and higher, but devices with AMD's Graphics Core Next 5 microarchitecture all showed rates under 1\%. This means that even from a single vendor, the behaviors of different architectures can vary widely and past results from one vendor cannot be counted on to predict future behaviors.

\subsection{MCS Bugs \label{sec:gpuharbor-vulkan-bugs}}

The conformance testing campaigns discovered bugs on several vendors' devices when running under the Vulkan and WebGPU frameworks.

\begin{enumerate}
    \item \textbf{Arm}: Coherence violations of the \textbf{MP-CO} litmus test were observed when using the Vulkan framework on two Arm GPUs, a Mali-G71 and a Mali-G78. These bugs were reported to and confirmed by Arm, leading to a compiler fix to insert a missing memory fence. Arm has also added regression tests based on the pattern of the reported violation.
    \item \textbf{NVIDIA}: Violations of the \textbf{MP-CO} test were also observed when using the Vulkan framework on an NVIDIA Tegra X1. Additionally, the WebGPU conformance test results revealed violations of a different read-read coherence test, \textbf{RR}, on an NVIDIA Quadro P620 running on a Linux desktop (therefore using Vulkan as the native framework). The combined report of the bug on the Tegra X1 and Quadro P620 helped NVIDIA find and fix a bug in its Vulkan compiler. NVIDIA also noted that this bug affected the Vulkan compiler in all pre-Volta architecture GPUs.
    \item \textbf{Apple}: The WebGPU conformance tests revealed coherence violations in \textbf{RR} on eight other devices, all running on Apple machines. Five of these devices were Intel GPUs, two were NVIDIA GPUs, and one was an AMD GPU. The same issue was observed and noted in \mychap\ref{chap:mc-mutants}. On Intel, it is likely that these bugs are instances of the same issue, but these results represent the first time the bug has been observed on non-Intel GPUs.
\end{enumerate}

All of the bugs were found by running conformance litmus tests under tuned system stress. The system stress was chosen using the MC Mutants approach, i.e., by selecting a system stress configuration effective at revealing weak behaviors in an associated weak memory litmus test. For the \textbf{MP-CO} bugs on the Arm and NVIDIA devices, a correlation analysis empirically validates the methodology.

Both the \textbf{MP} and \textbf{MP-CO} litmus tests were run in 150 randomly generated system stress configurations on each of the Android devices showing the bug, recording the rate of weak behaviors in \textbf{MP} and the rate of buggy (i.e., non-coherent) behaviors in \textbf{MP-CO}. The Pearson correlation coefficient (PCC) between the rate of weak and buggy behaviors is 0.732 on the Arm Mali-G71, 0.759 on the Arm Mali-G78, and 0.832 on the NVIDIA Tegra X1. Since these behaviors are recorded from 150 samples (i.e., system stress configurations), there are 148 degrees of freedom, and running a Student's t-test leads to a p-value less than $10^{-5}$ on each device. This shows that the PCC between weak behaviors and bugs is almost certainly not due to random chance, further validating that configurations tuned using weak behaviors are effective at revealing bugs in conformance tests.

All of the data was collected by running the tests with pseudo-randomly generated system stress configurations, but as mentioned in \mysec\ref{sec:gpuharbor-data-collection}, the generator is seeded with a known value.  For Apple and NVIDIA, the maximum similarity occurs between identical GPUs: two Apple M1 Maxes and two NVIDIA GeForce RTX 3080s. A similar analysis with Intel determines that an unknown device is most likely an Intel Iris Xe Graphics. While the primary interest is using these data to help choose conformance test strategies, as shown next, GPU MCS behavior data like this also exposes a fingerprinting risk, despite the specification trying to hide specific device information for security reasons.

\paragraph{Clustering-Based Testing Strategies} K-means clustering attempts to minimize the \textit{distortion}, or the sum of squared distances between vectors and a centroid. Applying k-means clustering to GPU MCS behavior has implications for testing strategies; when developing cross-platform GPU applications that rely on shared memory operations, testing these applications on a number of devices can increase confidence in the correctness of the implementation. A naive strategy might be to choose one device from each major vendor, but the results show that this is not necessarily an optimal strategy.

\mytablong\ref{tab:gpuharbor-webgpu-clusters} shows the result of running k-means with six clusters, representing each device by its cosine similarities to every other device. The ``elbow method'' heuristic showed that the rate of decrease in distortion leveled off at six clusters on the collected data. The clustering data shows that devices from the same vendor are generally placed into the same cluster, but there are outliers in each case. The only NVIDIA Kepler device in the study was dissimilar enough from other devices that it was placed in its own cluster. Kepler is also the oldest NVIDIA architecture in the study, showing that special testing attention might be needed when supporting older devices in cross-platform GPU frameworks.

When selecting which devices to test a shared memory GPU application on, the strategy should be to first choose a number of clusters based on the rate of decrease in distortion, and then select at least one device from each cluster. Using the collected data, this might mean selecting an AMD device from \textbf{A}, Apple devices from clusters \textbf{B} and \textbf{D}, NVIDIA devices from clusters \textbf{C} and \textbf{F} (the only device in that cluster), and an Intel device from cluster \textbf{E}. Despite choosing more Apple and NVIDIA devices than AMD and Intel, the similarity data ensures the tests maximally cover devices with different behavior profiles.

This subsection discusses a use case showing how the diversity of weak memory profiles across these different GPUs can impact software development. Locking algorithms are implemented using atomic operations to synchronize access to critical sections. Lock implementations depend on careful placement of memory fences to avoid compilers and hardware from reordering memory accesses, which can cause critical section failures. Three common spinlocks were implemented: test-and-set (TAS), test-and-test-and-set (TTAS), and compare-and-swap (CAS). Each of these locks specifically needs to disallow \textbf{MP} behaviors using acquire/release memory fences. However, the results in \mysec\ref{sec:gpuharbor-wb-char} show that on some mobile devices \textbf{MP} weak behaviors never occur, meaning that if the locks are tested on these devices, they may run correctly despite being incorrectly implemented according to the specification.

To investigate this, the three locks were tested on two Android devices, an Arm Mali-G78 and a Qualcomm Adreno 610. The locks were implemented both with and without appropriate acquire/release memory fences. In these tests, threads from different workgroups acquire the lock 10k times and increment a non-atomic memory location in the critical section. This test was run for 1k iterations, recording the number of critical section violations for each device and each lock.

On the Arm Mali-G78, a larger GPU which exhibits a relatively high rate of \textbf{MP} behaviors, critical section failures were observed in unfenced versions of all three locks; in every failure case except one, the value was 189,999 instead of 190,000, meaning that just one of the increments was not reflected. In the remaining failure case, the value was 189,998. On the Qualcomm Adreno 610, which exhibited no \textbf{MP} behaviors in the study, no failures were observed. Both devices exhibited no failures when locks were run with correct fences.

Therefore, when writing applications that require synchronization, care must be taken to ensure the application is tested on devices where incorrect implementations will lead to failures, highlighting the importance of collecting and characterizing MCS behavior data.

\section{Lessons Learned \label{sec:gpuharbor-lessons-learned}}

This section discusses important lessons learned while developing and running the study.

\paragraph{Ease of Use.}

Technical studies of low-level details like memory consistency specifications~\cite{alglave2011litmus,sorensen2016exposing,kirkham2020foundations} have been run by expert practitioners and involve installing special software (e.g., OpenCL/CUDA drivers) and running experiments from command-line interfaces. However, experiments that solicit non-technical users require accessible and frictionless interfaces in order to collect many results. Initially, users were asked to download their results and submit them directly by email, but many users would not take this seemingly small step. Thus, a way to submit results by simply clicking a button was implemented. This required substantial engineering effort, both to set up a client/server infrastructure and to distribute the tools in a non-technical way (e.g., through web browsers and app stores). Once implemented, this workflow also had the benefit of standardizing the experiments; instead of relying on users to configure their systems and choose the right options, all of this was built in so that users only had to click a few buttons to run and submit results.

\paragraph{Testing Time.}

Previous studies ran tests for hours or days, but it is unrealistic for volunteer users to run experiments that long on their devices. Therefore, the trade-off space between experiment time and behavior coverage was explored. Through trial and error, parameters were determined that allowed the collection of high-quality, standardized data in a short time frame, using testing techniques from \mychap\ref{chap:mc-mutants} that increased testing speed and provided statistical measures of reproducibility.

\paragraph{Enabling New Research Questions.}

Important research questions on memory consistency, including the three from \mysec\ref{sec:gpuharbor-wb-char}, require performing a large-scale study. For example, previous studies~\cite{kirkham2020foundations} have attempted to create portable testing strategies, but could only provide limited guidance on choosing representative sets of devices to test on due to the small number of devices in their evaluation. On the other hand, the collected data shows that GPUs from different vendors can behave similarly under stress, and thus portability may not be vendor-specific. Therefore, increasing the scale of evaluation through faster and more accessible testing should be an important factor when developing new testing strategies for a diverse and ever-growing set of devices.

\paragraph{Extensibility.}

When \textsc{LitGen} was first designed, it only generated SPIR-V shaders. However, when the focus expanded to testing WebGPU's MCS, \textsc{LitGen}'s neutral configuration language (JSON) made it straightforward to write a backend generator for WGSL shaders. Ensuring the tools are extensible means that they might also be useful for researchers testing other areas of GPU specifications, e.g., floating-point operation accuracy. In the same vein, the initial app only targets Android devices running Vulkan, but future development could expand its scope to support both Android and iOS devices.

\section{Investigating the MCS of NVIDIA GPUs \label{sec:gpuharbor-nvidia-mcs}}

The lessons learned while building a cross-platform MCS testing framework also apply to more focused investigations of a single vendor's hardware and programming framework. Although the MCS concepts encoded by a litmus test can be complex, each generated shader and its host-side setup are relatively small and well-defined; for example, a shader is less than 100 lines of code. The testing techniques developed for \gpuharbor were therefore translated to CUDA without redesigning the underlying approach. This port made it possible to test recent NVIDIA GPUs using the same systematic generation and execution strategies used throughout this chapter.

The resulting testing campaigns examine two aspects of NVIDIA GPU memory consistency. First, 560 variants of the message passing litmus test characterize synchronization on the GH200 under different memory scopes and fence placements. Second, 1,302 variants of classic litmus tests are run to investigate multi-copy atomicity and write propagation on A100, H100, GH200, and GB10 GPUs. These experiments contributed to the two projects discussed below. The discussion focuses on the testing contributions and results relevant to this thesis, while briefly placing them within the broader goals of each project.

\subsection{Consistency of NVIDIA's Hopper GPU Architecture \label{sec:gpuharbor-nvidia-grace-hopper}}

A language-level MCS is often weaker than the actual MCS supported by hardware, e.g., C++'s MCS is weaker than x86 TSO, in order to support multiple hardware platforms. Yet, a critical gap exists in the understanding of how closely NVIDIA GPUs correspond to the PTX MCS, which is important as compound MCSs like the one for NVIDIA's Grace-Hopper Superchip are developed. To provide empirical data of the MCS implemented by NVIDIA's Hopper GPU architecture, 560 variants of the \textbf{MP} litmus test were run on the GH200. \textbf{MP} is chosen because it represents the ubiquitous producer-consumer synchronization pattern found in many real-world applications.

The 560 variants represent all valid combinations of memory orders, memory scopes, and fence placements in the \textbf{MP} test. \mytablong\ref{tab:gpuharbor-mp-variants} provides an overview of how these variants were constructed. In CUDA terminology, the \textit{cta} scope corresponds to workgroup, \textit{gpu} corresponds to device, and \textit{system} is a scope unique to CUDA that synchronizes across multiple GPUs on the same system. In addition, CUDA supports SC fences, which in the \textbf{MP} test do not alter the semantics of the allowed outcomes compared to release-acquire fences, but these variants are still run for completeness.

Across all 560 litmus tests, whenever a release-acquire synchronization was established (i.e., both the release and acquire at \texttt{gpu} scope or higher), no weak behaviors were observed. Conversely, in all other scenarios---including insufficient scope and asymmetric cases where only one side had a release or acquire---weak behaviors were consistently observed. This confirms accurate enforcement of the PTX MCS and demonstrates that the microarchitecture implements scoped ordering aggressively: weak behaviors appear precisely and only when synchronization is absent.

These results played a key part in an overall project empirically investigating the consistency and coherency properties of the Grace-Hopper Superchip~\cite{bagchi2026gh}. The larger work looked at coherence and consistency of each component individually and combined, providing valuable empirical data and introducing new methods for reverse-engineering GPU coherence. This project was led by Soham Bagchi from the University of Utah.

\subsection{MCA and Write Propagation Properties of NVIDIA GPUs \label{sec:gpuharbor-nvidia-beyond-fence}}

The techniques in this thesis were also used to empirically characterize write propagation and determine whether recent NVIDIA GPUs exhibit non-multi-copy atomic behavior, with implications for the development of compound consistency models. As discussed in \mysec\ref{sec:bg-multi-copy-atomicity}, these properties constrain the behaviors that new sound models of compound consistency must permit. The investigation uses GPU-only PTX executions on three NVIDIA architectures and four machines: Ampere (A100), Hopper (H100 and GH200), and Blackwell (GB10).

Three tests, \textbf{2+2W}, \textbf{3.2W}, and \textbf{Z6.3}, measure how writes propagate between GPU threads. Six additional tests, \textbf{IRIW}, \textbf{ISA2}, \textbf{RWC}, \textbf{WRC}, \textbf{WRW+2W}, and \textbf{WWC}, test for non-MCA behavior. Each test was instantiated with combinations of memory orders up to, but not including, SC fences to determine which propagation behaviors remain observable with release and acquire synchronization. Variants also cover \texttt{cta} and \texttt{gpu} scopes and the possible cooperative thread array groupings for each test. For example, the four threads in \textbf{IRIW} can be assigned to cooperative thread arrays in 15 distinct ways. This systematic generation produced the 1,302 variants summarized in \mytab\ref{tab:gpuharbor-nvidia-propagation-tests}.

The tests use the CUDA port of the stress-tuning and mutation-testing techniques introduced earlier in this thesis. Relaxed variants first identify effective stress configurations, which are then used when executing strengthened variants. The broader testing campaign ran for 24 hours each on the A100, H100, and GB10 and for 6 hours on the GH200, producing more than 10 trillion executions across all tests.

Weak behaviors were common when all memory accesses were relaxed, which likely corresponded to thread-local reorderings. More importantly, a weak \textbf{IRIW} behavior was observed on the H100 even when its accesses were strengthened with release and acquire memory orders, which intuitively likely prevent thread-local reorderings. These strengthened observations occurred in fewer than one in a million executions, despite using stress configurations that readily exposed weak behaviors in relaxed variants.

Together, these results demonstrate that a model of NVIDIA GPU behavior cannot assume MCA, even for programs using release and acquire operations, and must accurately represent the hardware's write-propagation behavior. Because some permitted behaviors are extremely rare, effective empirical testing is necessary to avoid developing models that are stronger than their implementations. The results of this work informed the development of a new strengthening to the PTX MCS that supports independent compilation and composition of code by CPUs and GPUs sharing a cache-coherent interconnect. This project is currently under submission and is being led by Dennis Sprokholt from TU Munich.

The sensitivity of weak behavior observations raises the question of whether stress generated by one process affects weak behaviors observed by another. To investigate this question, the parallel MCS testing environment is divided between two processes. A \textit{listener} repeatedly executes litmus tests and records the number of weak behaviors, while a \textit{stressor} generates memory-system pressure. A signal exists when the listener observes a statistically significant increase in weak behaviors while the stressor is active. Experiments were run on three GPUs, shown in \mytab\ref{tab:gpuharbor-sidechannel-results}, as part of a larger investigation into MCS-based side-channels led by Sean Siddens (University of Washington) and Sanya Srivastava (Duke University).

Each trial compares repeated listener measurements with and without the stressor. The Common Language Effect Size (CLES) reports the probability that a randomly selected stressed measurement contains more weak behaviors than a randomly selected baseline measurement. A signal is classified as \textit{reliable} when every stressed measurement exceeds every baseline measurement. The effectiveness of this technique depends partly on how a GPU schedules kernels submitted by different processes. Under a \textit{sequential execution model}, the listener and stressor take turns executing on the GPU. The stressor may leave effects in shared hardware state, but it cannot directly contend with the listener while the listener executes. Under a \textit{parallel execution model}, kernels from both processes can execute concurrently, allowing the stressor to continuously affect the memory system while the listener measures weak behaviors.

\mytablong\ref{tab:gpuharbor-sidechannel-results} reports the percentage of trials producing any statistically significant signal, the percentage producing a reliable signal, and the increase in weak behavior observations under stress across three GPUs, detailed below.

\paragraph{AMD Radeon RX 7900 XT.}
The AMD GPU uses a sequential execution model. Nevertheless, the listener detected a signal in 13.8\% of trials, including reliable signals in 5.9\%. When signals occurred, stress increased weak behavior observations by 7,890\% on average and by as much as 170,300\%. These results show that effects persisting between sequentially executed kernels can create an MCS-based side-channel, although signals are relatively infrequent.

\paragraph{NVIDIA GeForce RTX 4070.}
The NVIDIA GPU also uses a sequential execution model by default. Signals occurred in 5.6\% of trials and were reliable in 2.8\%, making them less frequent and reliable than on the AMD GPU. However, their magnitude was larger: stress increased weak behavior observations by 27,768\% on average and by as much as 499,600\%.

\paragraph{Apple M3.}
The Apple M3 GPU uses a parallel execution model. This behavior was established using a microbenchmark in which two processes each execute a kernel that takes approximately $k$ time in isolation. When launched together, both kernels complete in approximately $k$, indicating concurrent execution.

Parallel execution made the M3 substantially more susceptible to the side-channel. Signals occurred in 52.4\% of trials and were reliable in 20.5\%, while stress increased weak behavior observations by 92,470\% on average and by as much as 4,921,300\%. Building on these results, the listener and stressor were configured to transmit information through the presence or absence of memory-system stress, creating a reliable proof-of-concept channel with a capacity of 16 bits per second.

While not included in this original study, later pilot experiments also showed that a similar side-channel could be created when multiple partitions of NVIDIA's MIG technology ran the listener and stressor processes. These initial results were then developed into a more comprehensive side-channel and set of attacks in a project that was led by Cheng Gu from the University of Rochester~\cite{gu2026behindbars}.

\section{Summary and Broader Impact \label{sec:gpuharbor-broader-impact}}

The MCS testing techniques developed in this thesis have provided the community with practical methods for understanding increasingly complex GPU memory systems. Stress tuning and mutation testing make rare weak behaviors easier to expose and provide evidence that a testing environment is effective. Portable test generation and accessible interfaces make these techniques usable across programming frameworks and devices from many vendors. Together, these contributions have helped find bugs in deployed implementations, informed official conformance tests, produced empirical data for developing new MCSs, and made GPU memory consistency testing available to researchers and developers beyond a small group of specialists. The accessibility of \gpuharbor has also made it useful for teaching, and it has been used in guest lectures, classes, and research presentations to help students learn concepts around memory consistency by exploring weak behaviors on real GPUs.

The work in this thesis was also applied in a cross-research group setting, leading to an article being published on SIGARCH's Computer Architecture Today blog~\cite{basu2025gpu-mcs-blog}. The article describes a feedback loop in which formal tools determine whether an optimization is sound according to an MCS, while empirical tools such as \gpuharbor test how the optimization behaves on existing hardware. In its case study, formal analysis identified an unsound synchronization optimization, while rigorous testing clarified when its unexpected behavior could be observed and exposed the role of a control dependency not captured by the model. This example demonstrates that empirical testing, when combined with formal analysis, can evaluate proposed optimizations, reveal gaps between specifications and implementations, and guide future performance tooling and MCS development.
