\section{GPU Architecture}
\label{sec:bg-arch}

\begin{figure}
\begin{center}
\includegraphics[scale=0.75]{images/gpu_arch.pdf}
\end{center}
\caption{Simplified GPU architecture. Cores (outlined blue squares) are organized hierarchically into subgroups (SG), with a Compute Unit (CU) consisting of several subgroups that share access to a common L1 cache. A shared-memory region ($S$) is managed by software to communicate between threads on the same CU without going through main memory. All CUs share access to a global L2 cache.}
\label{fig:bg-gpu-arch}

Numerous vendors make GPUs today, including NVIDIA, AMD, Arm, Apple, Qualcomm, and Intel, to name a few. Despite architectural and feature differences, many GPUs share a set of common abstractions. \myfiglong\ref{fig:bg-gpu-arch} shows a simplified view of a general GPU architecture. Unlike most CPUs, which have a smaller number of very powerful cores, GPUs trade flexibility and complexity of individual cores, e.g., complex branch predictors and out-of-order processing, in order to dramatically increase the number of threads that can execute in parallel. For example, a datacenter-level NVIDIA B200 GPU can execute over 300,000 threads in parallel~\cite{nvidia_blackwell_tuning_guide}, while a consumer-level Apple M5 GPU can execute over 10,000~\cite{apple_m5_announcement}.

Inside a GPU, processing elements, or cores, are organized into hardware-defined subgroups, with a typical size being 32, that execute in a single-instruction, multiple-thread (SIMT) manner. These subgroups each have their own allocated register space, and on many vendors' GPUs can perform special group-wise operations without having to communicate through the overall memory hierarchy. Cores on the same compute unit share access to an L1 cache. Alongside the L1 cache (sometimes even just a logical partition of the same physical memory), is the \textit{shared memory} region (represented by $S$ in \myfig\ref{fig:bg-gpu-arch}), which is managed by software and allows cores on the same CU to communicate without paying the cost of L2 memory access times. Additionally, there is an L2 cache that is shared across all CUs. The L2 is the final cache in the hierarchy, and communicates directly with main memory. On many smartphones and consumer-level laptops, the GPU is \textit{integrated}, sharing main memory with the CPU and other accelerators, while larger or datacenter-specific GPUs often have \textit{discrete} main memory that is connected to the CPU or other accelerators via a high-speed interface like PCIe.

Recent datacenter GPUs from vendors like NVIDIA and AMD are often very large, and a single user may not need to utilize the full GPU continuously. To enable more fine-grained allocation of GPU resources, NVIDIA has developed its Multi-Instance GPU (MIG) technology, which splits a GPU into several partitions. For security reasons, these partitions cannot share any user-visible micro-architectural state, as that would allow a malicious user of one partition to interfere with or infer characteristics of an application running on a separate partition. However, the work described in \mychap\ref{chap:gpuharbor} shows that in early implementations of NVIDIA MIG, this partitioning was not fully complete, allowing behavior to leak across the MIG partition boundary.

\section{GPU Programming}
\label{sec:bg-sw}

GPU frameworks consist of a host API that executes on the CPU and a programming language that executes on the GPU. Applications allocate resources, transfer data, and schedule GPU work through the host API, while programs are dispatched asynchronously to run on the GPU. Although GPU frameworks remain central to graphics workloads, this thesis focuses on GPGPU applications.

As much of this thesis is focused on cross-platform GPU programming, it mostly adopts terminology from a GPU framework called WebGPU~\cite{webgpu}. WebGPU is designed to run on GPUs from many vendors by targeting intermediate frameworks depending on the operating system it is running on. For example, WebGPU runs through Apple's Metal~\cite{apple_metal} framework on macOS and iOS, Vulkan~\cite{vulkan} on Linux and Android, and DirectX~\cite{directx} on Windows. The programming language for WebGPU is called the WebGPU Shading Language (WGSL)~\cite{webgpu-sl}, with the term \textit{shading} originating from GPU's early primary use for graphics processing. In WebGPU, a GPU program is called a \textit{shader}, which corresponds to the term \textit{kernel} in other popular GPU frameworks such as CUDA~\cite{cuda}.
As WebGPU is primarily designed to run in the browser, several runtimes are currently being developed for it: Dawn~\cite{dawn} for Chromium-based browsers, wgpu~\cite{wgpu} for Firefox, and WebKit~\cite{WebKit} for Safari.

The GPU execution model is hierarchical, with several execution scopes and memory address spaces. The base unit of execution is a \textit{thread}. Threads execute a single stream of instructions and are scheduled onto GPU processing elements by the implementation. A set of threads, with the size defined by the hardware (typically 32 or 64), is called a \textit{subgroup}, as mentioned in \mysec\ref{sec:bg-arch}. While subgroups are important for increasing the performance of modern GPU programs, especially in machine learning, this thesis does not investigate the specific behavior of subgroup operations, instead focusing on memory consistency independent of whether threads belong to the same subgroup. Threads are further organized into \textit{workgroups}; the programmer specifies the number of threads in a workgroup subject to an implementation-defined limit, typically no more than 1024. All threads in a workgroup have access to a software-managed shared-memory region, which is often a partition of the L1 cache and is called the \textit{workgroup} address space in WebGPU.

A group of workgroups launched at the same time is called a \textit{dispatch}. The total dispatch size can be larger than the number of threads hardware can execute in parallel, and the GPU is responsible for scheduling and running workgroups on the hardware as resources become available. All threads in a dispatch can read and write to global memory, which WebGPU calls the \textit{storage} address space. Global memory buffers are allocated by the host and bound to the shader, and additional memory cannot be dynamically allocated during shader execution. WebGPU also exposes a special \textit{uniform} address space of read-only memory, but as read-only memory does not require special consideration when it comes to memory consistency, this thesis does not investigate it.

\subsection{An Example Program}
\label{sec:bg-wgsl-example}

With an understanding of the overall GPU architecture and programming model, an example of a real WebGPU program is now described. \myfiglong\ref{fig:bg-wgsl-example} shows a WGSL shader that implements the same message passing idiom as shown in \myfig\ref{fig:intro-mprlx}. The shader contains one memory region, \texttt{buf}, which lives in the \texttt{storage}, i.e., global, address space and is both readable and writable. The size of this buffer is determined when it is allocated and bound to the shader before dispatch. This shader is specified to be a \texttt{compute} shader, and is launched with one thread per workgroup. The host side specifies the number of workgroups; in this example, the shader would be launched with two workgroups. Each thread can access its position in the overall grid of threads using the \texttt{global\_invocation\_id} builtin value. The grid can be launched with threads in up to three dimensions, but as this is just a notational difference this thesis uses only one-dimensional grids.

Thread 0 runs the producer side of the message passing idiom, writing 1 to \texttt{buf[0]}, i.e., the data, and then writing 1 to \texttt{buf[1]}, i.e., the flag. To avoid data races, all memory operations are atomic. Thread 1 runs the consumer side of the message passing idiom, reading the flag and then the data. Because the threads execute concurrently, the order in which they run is non-deterministic. The allowed outcomes depend on the WebGPU MCS; in this program all of the outcomes from \mytab\ref{tab:intro-mp-orderings} may occur. To understand more precisely how the memory operations in programs like this one can be reasoned about, memory consistency specifications are now formally introduced.

\section{Memory Consistency Specifications}
\label{sec:bg-mcs}

A memory consistency specification (MCS), also known as a memory model, defines the allowed behaviors of shared-memory concurrent programs. Several strategies are used to formalize MCS semantics, including axiomatic, operational, and denotational models. Official and formal models for GPU programming frameworks like CUDA, Vulkan, and WebGPU describe their models axiomatically, so this thesis uses axiomatic models exclusively. In an axiomatic MCS, each possible set of outcomes for a program's reads and writes is called a \textit{candidate execution} consisting of a set of events and relations between those events~\cite{alglave2014herding,batty2011mathematizing}. The MCS then classifies each candidate execution as consistent, and therefore allowed, or inconsistent, and therefore disallowed.

\subsection{Memory Events, Orders, and Scopes}

The events, relations, and annotations used in this thesis are shown in \mytab\ref{tab:bg-mcs}. Events considered in this thesis are reads ($R$), writes ($W$), atomic read-modify-writes ($RMW$), and fences ($F$). Reads obtain values from memory, writes place values in memory, and an RMW indivisibly reads and updates a memory location. Reads and writes may be atomic or non-atomic, while RMWs are necessarily atomic. A fence does not directly access a memory location; instead, it constrains the ordering or visibility of memory operations that occur around it.

Atomic operations and fences may be annotated with a \textit{memory order}. An atomic operation with a \textit{relaxed} memory order happens indivisibly, but does not synchronize operations on different threads. A \textit{release} write operation, or release fence that is followed by an atomic write, makes preceding operations visible to a thread that performs a corresponding \textit{acquire} read operation, or has an acquire fence preceded by an atomic read operation, and reads the value stored by the write operation. Acquire and release operations therefore support message passing programs such as the one shown in \myfig\ref{fig:intro-mprelacq}. A sequentially consistent (\textit{seq\_cst}, or SC) operation enforces acquire and/or release semantics and additionally participates in a single order of SC operations that is consistent with the other constraints of the MCS.

GPU frameworks may also annotate atomic operations and fences with a \textit{memory scope}, which restricts the threads to which their ordering and visibility guarantees apply. Two scopes relevant to this thesis are the \textit{workgroup} scope, which applies only to threads in the same workgroup, and the \textit{device} scope, which applies to all threads in the same dispatch. Release/acquire operations can synchronize only when their scopes include the participating threads.

\subsection{Relations Between Events}

Events from the same thread are related through program-order ($po$), and $\textit{po-loc}$ is the restriction of $po$ to events that access the same memory location. Every read and RMW is related to the write or RMW whose value it observes through the reads-from ($rf$) relation. Writes and RMWs to the same atomic location are ordered through coherence order ($co$). The from-read ($fr$) relation is derived from $rf$ and $co$: it relates a read or RMW $r$ to a write or RMW $w$ when $r$ reads from a write $w'$ that precedes $w$ in coherence order. The communication relation ($com$) is the union of $rf$, $co$, and $fr$.

Synchronizes-with ($sw$) relates release and acquire operations that synchronize across threads. For direct release/acquire atomics, a release write or RMW synchronizes with an acquire read or RMW that reads its value. Fences can provide the same synchronization around relaxed atomic operations. For the fence-based synchronization used in this thesis, an atomic write or RMW $w$ synchronizes with an atomic read or RMW $r$ if $w$ is preceded, not necessarily immediately, by a release fence, $r$ is followed, also not necessarily immediately, by an acquire fence, $r$ reads the value written by $w$, and the fences' scopes include both threads.

The happens-before relation ($hb$) is the transitive closure of $po \cup sw$. It captures both ordering within a thread and ordering established by synchronization between threads. In the subset of MCS behavior considered in this thesis, candidate executions must have an acyclic $hb$, and reads must not observe a write that is older than another write ordered before the read by $hb$.

SC operations impose ordering beyond the release/acquire-derived $hb$ relation by participating in an SC order. Defining this order correctly is subtle; C++ and CUDA describe SC ordering differently~\cite{batty2011mathematizing,lustig2019ptx}, and prior work has identified flaws in the original C++ and OpenCL SC definitions~\cite{batty2016overhauling,lahav2017repairingc11}. The frameworks and behaviors considered in this thesis either do not support or do not require SC atomics, so this thesis does not otherwise formalize their ordering.

\subsection{Coherence and Data Races}

Coherence constrains the behavior of accesses to each atomic memory location. The models used here follow existing literature, adopting the SC-per-location axiom which states that coherence requires the union of $\textit{po-loc}$ and $com$, restricted to atomic operations on a location, to be acyclic~\cite{alglave2014herding}. Intuitively, threads may observe operations on different atomic locations in different orders, but their observations of operations on any one atomic location must be compatible with a single order.

A data race occurs when two threads perform conflicting non-atomic accesses to the same memory location, at least one access is a write, and neither access happens before the other. Unlike atomic operations, these accesses do not receive coherence guarantees, and without a synchronizes-with edge between the threads there may be no cross-thread $hb$ ordering. In languages such as C++, a data race can invalidate a program and cause it to fall outside the guarantees of the MCS~\cite{batty2011mathematizing}, while languages such as Java provide very weak semantics for data races~\cite{manson2005java}. Some GPU frameworks like Vulkan also leave data races as undefined, while WebGPU classifies data races as \textit{dynamic errors}; their effects are indeterminate, but they should not lead to security vulnerabilities. This thesis focuses on testing programs in which all shared-memory accesses are atomic, called \textit{trivially data-race-free} (TDRF) programs, while the memory safety analysis in \mychap\ref{chap:saferace} assumes that programs with any non-atomic accesses may include data races.

\subsection{Multi-Copy Atomicity \label{sec:bg-multi-copy-atomicity}}

The valid behaviors of an MCS can be described by how threads reorder memory accesses locally, and by how memory writes become visible to the rest of the system~\cite{arvind2006memmodel}. This second property is called \textit{multi-copy atomicity} (MCA) or \textit{write atomicity}. In an MCA system, threads may observe two independent writes in either order, but all threads must agree on that order. In a non-MCA system, a write may propagate to different threads at different times, allowing the threads to disagree on the order in which the writes became visible.

The independent reads of independent writes (IRIW) litmus test in \myfig\ref{fig:bg-iriw} illustrates this distinction. Threads 0 and 1 independently write to \texttt{x} and \texttt{y}. Thread 2 observes \texttt{x} before \texttt{y}, while thread 3 observes \texttt{y} before \texttt{x}. In the weak outcome, thread 2 observes the write to \texttt{x} but not the write to \texttt{y}, while thread 3 observes the write to \texttt{y} but not the write to \texttt{x}. The observers therefore disagree about which write occurred first.

One important note here is that non-MCA behavior can only be confirmed if it is guaranteed that thread-local reorderings of the two reads in each of thread 2 and 3 are prohibited. At the hardware level, CPU architectures like x86 and ARMv8~\cite{flur2016arm} support MCA. In x86, loads are not reordered by hardware, while in ARMv8 a memory barrier inserted between the two loads prevents the reordering. However, given the MCS relations described so far, acquire fences like the ones in \myfig\ref{fig:bg-iriw} are not sufficient to disallow the non-MCA behavior, as the axiomatic MCS description does not directly model propagation of writes. Therefore, language-level axiomatic MCSs, including for C++ and GPU languages, generally do not model MCA behavior, and the write propagation properties of GPUs have not been rigorously studied.

One option for investigating the MCA properties of hardware is to add fences to the high-level program, as shown in \myfig\ref{fig:bg-iriw}, with the assumption that the fence will prevent the thread-local reordering. While this assumption is not formalized, testing it can reveal characteristics about how modern GPUs behave, informing the development of new MCSs that more accurately model hardware.

\subsection{GPU MCSs}

\paragraph{WebGPU.}
This thesis considers the subset of WebGPU's MCS needed to reason about the programs and tests used in later chapters. WebGPU atomic operations are given a default memory order of \texttt{relaxed} and cannot be directly annotated with other memory orders. Atomic operations on workgroup memory have \texttt{workgroup} scope, while atomic operations on device memory have \texttt{device} scope.

A \textit{control barrier} requires all participating threads to reach the barrier before any may continue, while a \textit{memory fence} constrains the ordering and visibility of surrounding memory operations without itself requiring threads to rendezvous. WebGPU does not expose standalone fence instructions. Instead, \texttt{workgroupBarrier} and \texttt{storageBarrier} combine control-barrier and memory-fence semantics. Both have a control and memory scope of \texttt{workgroup}; \texttt{workgroupBarrier} applies to workgroup memory, while \texttt{storageBarrier} applies to device memory. Consequently, WebGPU provides no way to establish release/acquire synchronization between threads in different workgroups. Earlier versions of the WGSL specification described barrier operations as providing release/acquire semantics across workgroups, but, as discussed in \mychap\ref{chap:mc-mutants}, testing helped show that this behavior was too strong for WebGPU's backend constraints at the time.

\paragraph{Vulkan.}
Vulkan, one of WebGPU's target frameworks, exposes a richer memory model through SPIR-V~\cite{spir-v}. Atomic operations and memory barriers can directly specify memory orders and scopes, including acquire and release operations with device scope.  Vulkan's additional operations support synchronization patterns that WebGPU cannot express, including release/acquire synchronization between workgroups. As the WebGPU MCS is a subset of the Vulkan MCS, it can be mapped directly onto it.

\paragraph{Other MCSs.}
WebGPU also targets Apple's Metal GPU framework, which includes an MCS that is based on the C++ MCS, but like WebGPU only allows atomic operations to be relaxed. Recently, Metal added support for fences that support inter-workgroup release/acquire memory semantics~\cite[\mysec 6.15.1]{metal-sl}, which it did not support when WebGPU's MCS was originally defined. WebGPU also targets DirectX, which does not specify a formal MCS but which WebGPU implementations are able to target by following the informal rules around special \textit{interlocked} operations in DirectX's HLSL programming language.

NVIDIA's CUDA framework also includes a formal MCS for its intermediate representation, PTX, which includes both release/acquire and sequentially consistent semantics. For the purposes of this thesis, only the relations and terminology described in \mysec\ref{sec:bg-mcs} are required.

\section{Litmus Tests and Empirical MCS Testing}
\label{sec:bg-litmus}

Litmus tests are small concurrent programs used to illustrate~\cite{shasha1988efficient}, compare~\cite{wickerson2017automatically,madorhaim2010generating}, and empirically test~\cite{alglave2011litmus,alglave2015gpu,kirkham2020foundations} MCSs. A litmus test has many candidate executions, and an execution is identified by recording the values returned by reads and checking the final state of memory. In some tests, it may not be possible for an outside observer to infer the order in which writes occur, i.e., if they are overwritten during the program execution. In these cases, an additional observer thread may be added to infer coherence order by recording the order in which writes become visible. Litmus tests are often used in MCS documentation~\cite{batty2011mathematizing} and can also be used to derive or compare formal models~\cite{bornholt2017memsynth}.

\myfiglong\ref{fig:bg-mp-weak-execution} depicts the weak candidate execution of the fence-free message passing (MP) test from \myfig\ref{fig:intro-mprlx} using relaxed atomic operations. In thread 0, \introMpEventA is related to \introMpEventB by program order; in thread 1, \introMpEventC is related to \introMpEventD by program order. The read of \texttt{flag} at \introMpEventC reads from the write at \introMpEventB, producing an $rf$ edge from \introMpEventB to \introMpEventC. The read of \texttt{data} at \introMpEventD reads from the initial write of 0 rather than from \introMpEventA. Because the initial write precedes \introMpEventA in coherence order, this produces an $fr$ edge from \introMpEventD to \introMpEventA. The program contains no $sw$ edges, so the $hb$ relation contains only the $po$ edges within the two threads and does not order \introMpEventA before \introMpEventD. The candidate execution is therefore consistent with the MCS, allowing thread 1 to observe the new value of \texttt{flag} while reading the old value of \texttt{data}.

\myfiglong\ref{fig:bg-mp-fenced-execution} depicts the corresponding candidate execution of the fenced MP test from \myfig\ref{fig:intro-mprelacq}. The release fence before \introMpEventB and acquire fence after \introMpEventC cause the write at \introMpEventB to synchronize with the read at \introMpEventC when \introMpEventC reads the value written by \introMpEventB. This adds the edge $\introMpEventB \xrightarrow{sw} \introMpEventC$ between the write and read, rather than between the fences themselves. Because $hb$ is the transitive closure of $po \cup sw$, the path
\[
    \introMpEventA \xrightarrow{po} \introMpEventB
    \xrightarrow{sw} \introMpEventC
    \xrightarrow{po} \introMpEventD
\]
places \introMpEventA before \introMpEventD in $hb$. The read at \introMpEventD must therefore observe the value written by \introMpEventA rather than the initial value, making the depicted weak candidate execution inconsistent with the MCS.

Although the fence-free weak behavior is allowed by the specification, it may still be rare on a particular implementation because its empirical occurrence depends on compiler transformations, hardware behavior, scheduling, caches, and the surrounding test environment. In particular, the empirical rate of observed weak behaviors is highly sensitive to system stress, and prior work has shown that GPU MCS testing requires carefully tuned stress to expose rare behaviors~\cite{alglave2015gpu,sorensen2016exposing,kirkham2020foundations,levine2023mcmutants}. Early GPU application work did not observe weak behaviors even when specifications allowed them~\cite{feng2010synchronize}, while later work showed that specialized system stress can reveal many surprising weak behaviors~\cite{alglave2015gpu,sorensen2016exposing}.

Executing litmus tests on deployed systems can be used for two purposes, which is illustrated using a litmus test $L$ that can exhibit a weak behavior execution $e$, and an MCS $S$.
\begin{enumerate}
    \item \textbf{Conformance testing}: if $e$ is \textit{disallowed} in $S$, then implementations of $S$ can be checked. That is, if a platform $p$ claims to implement $S$, then $L$ can be executed many times on $p$, checking for $e$. The observation of $e$ would indicate a bug.
    \item \textbf{Profiling weak behaviors}: if $e$ is \textit{allowed} in $S$, and a platform $p$ claims to implement $S$, then $L$ can be executed many times on $p$ to understand the extent to which that platform allows $e$. In some cases, $p$ might not show $e$ empirically, or $e$ may appear more frequently under a certain configuration of system stress. A collection of this type of data creates a weak memory profile for $p$.
\end{enumerate}

The tests primarily used in this thesis include the classic two-thread, four-instruction litmus tests that expose pairwise memory reorderings. For example, store-buffering tests expose write-read reorderings, load-buffering tests expose read-write reorderings, and the message-passing test exposes write-write or read-read reorderings. These tests are small, but they capture synchronization patterns used in common concurrent algorithms and have been useful in finding both application bugs and MCS implementation bugs~\cite{sorensen2016exposing,kirkham2020foundations}. In \mychap\ref{chap:gpuharbor}, three and four-thread tests like the IRIW one in \myfig\ref{fig:bg-iriw} are also used to investigate the MCA properties of NVIDIA GPUs.

\section{Memory Safety and Data Races in WebGPU}
\label{sec:bg-mem}

Another important consideration is how languages like WebGPU maintain memory safety when shaders contain data races along with other sources of dynamic errors. Browser-exposed GPU APIs run code from potentially untrusted sources and must preserve isolation between different applications, tabs, and GPU allocations. Some browsers, including Chrome, run GPU commands from multiple tabs in one GPU process, which means that GPU memory-safety guarantees cannot simply rely on ordinary operating-system process isolation~\cite{chromium_webgpu_report}.

In WGSL, memory locations are either atomic or non-atomic, and memory locations cannot be cast from one kind to the other as they can in languages such as C++. Memory operations access sets of 8-bit memory locations. If two threads perform non-atomic operations on an overlapping set of memory locations, at least one operation is a write, and the accesses are not properly synchronized, then a data race occurs. Threads may concurrently perform atomic operations on the same atomic memory location without causing a data race. WebGPU's memory layout restrictions also prevent the mixed-size and non-aligned atomic accesses that can complicate race definitions in other settings~\cite{flur2017mixed}. These restrictions simplify the relationship between atomic locations and non-atomic locations compared to lower-level languages, where the same bytes may sometimes be accessed through different types or at different offsets.

Atomic operations, along with special synchronization operations, e.g., workgroup barriers, can be used to avoid data races and constrain allowed behaviors. However, data races are a global property, so a race \textit{anywhere} in a program poses risks. Therefore, when analyzing the security of an overall program in the presence of data races this thesis considers only synchronization-free programs, which may include non-atomic or relaxed atomic operations but do not include synchronization-inducing barriers or atomics.

While relaxed atomic operations cannot be used on their own to synchronize threads, they do provide some guarantees. A relaxed atomic operation must happen indivisibly, i.e., it cannot be split into separate operations. The value of a relaxed atomic read may not be assumed, prohibiting optimizations such as rematerializing a read from memory if the compiler has already optimized based on some value for that read. These operations and restrictions apply in both WGSL and in the languages that WebGPU targets.

Shaders written in WGSL can contain behaviors that are normally undefined in lower-level languages, including data races, uninitialized accesses, out-of-bounds accesses, and undefined arithmetic. WebGPU gives defined semantics to uninitialized accesses and undefined arithmetic. Other operations, including data races and out-of-bounds accesses, are classified as \textit{dynamic errors}. If a dynamic error occurs during shader execution, WebGPU specifies that memory accesses should affect only memory allocated for that shader~\cite[\mysec 7.3]{wgslspec}.

WGSL ensures memory safety by either (1) relying on underlying framework guarantees; (2) preemptively clamping memory accesses; or (3) inserting checks that drop them if they would occur out-of-bounds. For example, DirectX guarantees that out-of-bounds reads on device memory must return 0, while out-of-bounds writes are discarded~\cite{microsoft2024d3d11csaccess}. Vulkan provides an optional feature called \textit{robust buffer access} that enables the same protections on device memory at some cost to performance~\cite{vulkan2024robustness}. However, both DirectX~\cite{ldraw2024} and Vulkan~\cite[\mysec 3.52.8]{spir-v} leave the behavior of out-of-bounds accesses in workgroup and private memory undefined. Metal, by not describing their behavior, implicitly leaves all out-of-bounds accesses undefined~\cite{metal-sl}.

In a program free of data races, the combination of underlying guarantees and inserted bounds checks are enough to provide memory safety. However, the presence of data races raises additional complications. In fact, one of the possible outcomes of clamping an out-of-bounds access \textit{is a data race}, as the clamped index, which is now in-bounds, may already be accessed by some other thread. Therefore, it is necessary to consider the effects of data races on all programs with potential out-of-bounds accesses.

While WGSL classifies data races as dynamic errors, the lower-level frameworks it targets often treat data races as undefined or unspecified. Vulkan's formal memory model requires programs to be data-race free to be well-defined~\cite[App. B]{vulkan}. Metal is based on C++14, where data races lead to undefined behavior~\cite{cppreference2024multithread}. DirectX does not describe the consequences of data races, implicitly leaving their behavior undefined. This mismatch is a specification problem for WebGPU, as WGSL aims to provide memory safety even in the presence of dynamic errors, but the compilation stack may pass through languages and optimizers that do not preserve meaningful semantics for racy programs.

\section{Related Work}
\label{sec:bg-related-work}

This section first discusses GPU memory semantics and validation (\mysec\ref{sec:bg-related-semantics}), followed by GPU memory safety, isolation, and side-channels (\mysec\ref{sec:bg-related-security}). Lastly, it discusses general testing methodologies that the GPU-specific methodologies in this thesis build on (\mysec\ref{sec:bg-related-methodologies}).

\subsection{GPU Memory Semantics and Validation}
\label{sec:bg-related-semantics}

\paragraph{Formal MCSs.} Formal GPU MCSs, including models for OpenCL, SPIR-V, and PTX, make it possible to derive and classify litmus tests systematically~\cite{batty2016overhauling,vulkan,lustig2019ptx}. Prior work has generated litmus tests from formal MCSs~\cite{alglave2010fences}, studied the test sizes required to distinguish MCSs~\cite{madorhaim2011litmus}, synthesized comprehensive test suites~\cite{lustig2017automated}, and used litmus tests to compare MCSs~\cite{wickerson2017automatically,madorhaim2010generating}. This thesis instead uses formal MCSs as specifications for empirical validation, identifying the behaviors that tests should expose and providing a basis for evaluating testing methodologies. The thesis does not introduce new formal MCSs or prove their properties.

\paragraph{Empirical MCS Testing.} Work on testing MCS dates back to tools like ARCHTEST~\cite{archtest} and TSOTool~\cite{hangal2004tsotool}, which each generated test programs containing sequences of loads and stores and then looked for violations of sequential consistency. With the introduction of formal MCSs, researchers developed tools like \textsc{litmus}~\cite{alglave2011litmus}, which runs litmus tests generated from formal models directly on ISAs (namely x86, Power, and Arm) and includes stress parameters that make weak behaviors more likely. To reduce the overhead of running a single test iteration, and evaluating its outcome before running another iteration, techniques like PerpLE speed up MCS testing by executing algebraic sequences that can be analyzed once after many iterations~\cite{melissaris2020perple}. Techniques for CPU MCS testing have been extended to GPUs~\cite{alglave2015gpu,kirkham2020foundations}. Other GPU validation approaches automatically generate data-race-free programs to test whether implementations preserve expected program behavior~\cite{ta2019autonomous}. In contrast, the approaches in this thesis use small litmus tests to target specific MCS behaviors and conformance properties. Weak behaviors on GPUs are notoriously difficult to reveal, leading to work that statistically analyzed tuning techniques and reproducibility of results when running litmus tests on GPUs~\cite{kirkham2020foundations}.

\paragraph{Data Races.} Prior work on data races has tried to classify races as either ``benign'' or ``destructive''~\cite{narayansamy2007benign,flanagan2010destraces}, or argued that there are no benign races~\cite{boehm2011benign} and shown how races can break basic assumptions of program behavior~\cite{dolan2018bounding}. Other work has developed memory models specifically for heterogeneous systems, including heterogeneous-race-free models~\cite{hower2014hrf} and semantics for relaxed atomics~\cite{sinclair2017rats}. There is a large line of research on detecting and avoiding races, both statically~\cite{pratikakis2011locksmith,flanagan2001static,blackshear2018racerd} and dynamically~\cite{flanagan2009fasttrack,serebryany2009tsan,savage1997eraser}. Researchers have also developed tools to statically analyze GPU programs for data races~\cite{betts2012gpuverify,cogumbreiro2021compdrfgpu,li2010gpusmtverify}, but these tools are not complete and may report false positives, which hamper adoption by industry~\cite{johnson2013nostatic,sadowski2018staticgoogle,nachtigall2019explainingstatic}. More recently, a sound and partially-complete data race checker for GPUs has been developed~\cite{liew2024dataraces}, and a GPU programming language which prohibits data races and out-of-bounds accesses by construction has been proposed~\cite{kopcke2024descend}. The work in \mychap\ref{chap:saferace} does not attempt to detect or remove data races; instead, it focuses on providing guarantees that data races will not compromise the security of a programming language.

A stronger semantics for data races has been proposed, called \textit{Local DRF}, which specifies that data-race-free portions of programs have sequential semantics~\cite{dolan2018bounding}. While Local DRF provides compositional reasoning, it also forbids behaviors that have been observed on GPUs, like the load buffering behaviors described in \mychap\ref{chap:gpuharbor}. %Our pilot experiments also showed that the effects of data races can cross synchronization boundaries on GPUs, further hampering its adoption for WebGPU.

\subsection{GPU Security}
\label{sec:bg-related-security}

\paragraph{Memory Isolation.} Memory isolation on GPUs has been extensively studied---CUDA was shown to leak global and register data~\cite{dipietro2016cudaleaks}, leftover global memory from NVIDIA and AMD GPUs could be used to infer webpages visited by a user~\cite{lee2014stealing}, leftover shared and register memory could be used to reconstruct inference output of neural networks~\cite{sorensen2024leftoverlocals,pustelnik2024whispering}, and out-of-bounds accesses on NVIDIA GPUs could be used to overwrite threads' local memory and degrade output of neural networks~\cite{guo2024usenix}.

All this prior work also has focused on reverse engineering low-level details of GPU assembly, handwriting shaders to exhibit issues, or combinations of both to reveal specific, one-off vulnerabilities. In contrast, \mychap\ref{chap:saferace} presents a set of properties that guarantee memory isolation in a high-level language, WGSL, by preserving memory safety through the compilation stack.

\paragraph{Side-channels.} Memory-based side-channels include techniques that measure contention on storage devices on Linux systems~\cite{jiang2024sync}, utilize specific instructions to leak cache coherence states~\cite{guo2024usenix}, and analyze memory traffic of compressed data on GPUs to reconstruct images~\cite{wang2024gpuzip}. Other GPU-specific side-channels include techniques that rely on contention in the L1 and L2 caches~\cite{naghibijouybari2017constructing}, reverse-engineer other security features and use them as a side-channel~\cite{nazaraliyev2025refresh}, or spy on GPU cache activities~\cite{zhang2024invalidate}. However, none of this prior work analyzed how the MCS could be used as a high-level starting point for identifying and exploiting side-channels in GPUs, as the work in \mysec\ref{sec:gpuharbor-sidechannel} does.

\subsection{General Testing Methodologies}
\label{sec:bg-related-methodologies}

The testing and validation approaches in this thesis fall under several general areas: mutation testing, testing at scale, and compiler fuzzing.

\paragraph{Mutation Testing.} Mutation testing seeks to evaluate the efficacy of testing techniques. Originally designed as a form of white-box testing for sequential programs~\cite{demillo1978hints}, it has since been applied to concurrent programs while enforcing deterministic test outcomes~\cite{carver1993mutation}. Recent work has examined white-box mutation testing in the context of non-deterministic, flaky tests, using a test re-running strategy to increase the confidence in mutation scores~\cite{shi2019flaky}. The approach described in \mychap\ref{chap:mc-mutants} similarly addresses non-determinism; however, it evaluates testing environments rather than testing suites and targets memory consistency runtime platforms rather than parallel software.

Black-box testing is used to test a system without modifying its internal behavior. Black-box tests are usually derived from a system's specification, which led researchers to develop specification-based mutation testing~\cite{budd1985specification}, where test data is generated from a specification that has itself been mutated. Over the years, specification based mutation testing has been applied to predicate calculus, network protocols, finite state machines, and security policies~\cite{jia2011mutation}. The approach in \mychap\ref{chap:mc-mutants} builds on these foundations using well-developed MCS formalisms.

\paragraph{Testing at Scale.} Previous GPU MCS testing studies have been limited in the number of devices they were able to test. In contrast, the study in \mychap\ref{chap:gpuharbor} introduces tooling that allows the largest ever GPU MCS testing campaign to be conducted, running tests across 2 frameworks, 7 vendors, and \totalDevices devices. Other studies have tested relatively large numbers of devices, searching for bugs in compilers and hardware, although they have not been testing for MCS bugs specifically. For example, in~\cite{donaldson2017graphicsfuzz}, 17 GPU and driver combinations were tested for general GPU compiler bugs.

The approach described in \mychap\ref{chap:gpuharbor}, distributing the GPU MCS testing experiment using a web interface, is a form of \textit{volunteer computing}, where the general public volunteers their computing resources for research studies. Volunteer computing has been used for many compute-intensive tasks, including searching for extraterrestrial life~\cite{anderson2002seti}, training neural networks~\cite{desell2017large}, sequencing genomes~\cite{pellicer2005gene}, and climate modeling~\cite{christensen2005challenge}.

\paragraph{Compiler Fuzzing.} Compiler fuzzing has been widely studied, e.g., finding bugs in C compilers~\cite{yang2011csmith} and GPU compilers for languages like OpenCL~\cite{lidbury2015clsmith}, OpenGL~\cite{donaldson2017graphicsfuzz}, SPIR-V~\cite{donaldson2021spirvfuzz}, and even WGSL~\cite{lecoeur2023reconditioning}. These approaches are either based on differential testing~\cite{mckeeman1998differential}, comparing outputs of multiple compilers, or metamorphic testing~\cite{chen1998metamorphic}, where programs are changed in ways that should not affect their semantics and results are compared. Generally, these approaches explicitly seek to avoid undefined behavior, constructing programs that are guaranteed to not contain undefined behaviors or adding transformations that remove undefined behavior at the source level. Unlike these approaches, the compiler fuzzing in \mychap\ref{chap:saferace} explicitly includes constrained sources of undefined behavior, e.g., data races, out-of-bounds accesses, and undefined arithmetic, to test the security properties of compilers.
