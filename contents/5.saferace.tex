\section{Introduction}

This chapter presents \saferace, a thorough investigation into how data races can affect memory safety when compiling WGSL, a memory-safe language, to lower-level unsafe languages. \saferace includes threat assessments that test native GPU frameworks and WebGPU implementations for memory isolation vulnerabilities, along with specification proposals that preserve WGSL's memory safety in the presence of data races. These contributions are motivated by the identification of a data race specification vulnerability (\drsv) in the WebGPU framework, caused by a mismatch between the guarantees provided by WGSL and its compilation targets.

\subsection{WGSL Data Race Specification Vulnerability \label{sec:saferace-spec-vuln}}

In many languages, e.g., C and C++, data races cause undefined behavior, as they can lead to extremely counterintuitive outcomes. However, because these behaviors are not desirable in security-sensitive contexts (e.g., browsers), languages such as JavaScript and Wasm have defined data races, albeit with very permissive semantics~\cite{watt2019weakening,watt2020repairing}. Similarly, WGSL uses compiler passes and safe libraries to avoid many types of undefined behavior. However, the languages that WGSL targets are \textit{not} free of undefined behavior, and in particular follow C/C++ in giving data races undefined behavior.

\myfiglong{\ref{fig:saferace-oob-example-a}} shows a WGSL program with two arrays of unsigned integers, \texttt{i} and \texttt{m}, and two threads performing non-atomic accesses to these arrays. This program contains behaviors that, in isolation, are undefined in some languages: (1)~because memory locations are non-atomic and there is no synchronization, there are data races between ($a$, $c$) and ($b$, $c$), and (2)~while the race technically makes the program undefined, a possible natural compilation could allow for an out-of-bounds read if $b$ reads the value $c$ wrote to \texttt{i[0]}. The potential out-of-bounds read is a memory safety violation that, in turn, could lead to a vulnerability. The highlighted code in operation $b$, i.e., \textcolor{red}{\texttt{min(..., 3)}}, shows a check that WGSL compilers include to enforce memory safety. However, WGSL compilers currently contain no special protections against data races.

\paragraph{The Problem: Optimizing Away Bounds-Checks.} Even with the bounds-check inserted, a compiler for an unsafe lower-level language can optimize the code in a way that creates a ``time-of-check to time-of-use'' (TOCTOU)~\cite{abbott1976toctou} memory safety violation. Specifically, a compiler for an unsafe language is free to assume programs do not contain data races. Therefore, when only looking at the operations in thread 0, the compiler may notice that \texttt{i[0]} is set to 2 and remains unchanged until it is loaded in $b$. Based on a range analysis, it can determine that the call to \texttt{min} is unnecessary and remove it, leading to an out-of-bounds access if \texttt{i[0]} reads 4 in $b$. While it is unlikely that a real implementation would perform the problematic optimization on this particular program, \mysec\ref{sec:saferace-extended-example} describes a more realistic program that could trigger this optimization.

The program in \myfig\ref{fig:saferace-oob-example-a} straightforwardly demonstrates how data races can lead to vulnerabilities, but memory safety issues can occur due to other surprising behaviors allowed by data races. In particular, \myfig\ref{fig:saferace-oob-example-b} shows a WGSL program that contains the same data race between $a$ and $c$, but now the access to \texttt{m} in $b$ depends on a separate variable \texttt{x}. Prior work shows how reasonable implementations might set \texttt{x} to 4---for example, if the compiler notices that both \texttt{x} and \texttt{i[0]} contain the same value (2) and decides not to allocate a register for \texttt{x}, reloading \texttt{i[0]} instead and causing a potential out-of-bounds access~\cite{dolan2018bounding}.

To recap, WGSL code, i.e., \myfig\ref{fig:saferace-oob-example} without the \textcolor{red}{\texttt{min}}, is written by application developers and may contain out-of-bounds accesses and data races. To avoid out-of-bounds accesses, WGSL compilers insert bounds-checks when generating lower-level native GPU code. The WGSL specification also mandates that if a data race occurs, memory accesses should be restricted to in-bounds memory in order to maintain memory safety~\cite[\mysec 2.2]{wgslspec}, but WGSL compilers do not include any special protections against data races. However, native GPU languages currently give undefined semantics to data races, allowing the optimizations described above when compiling to vendor-specific machine code. This mismatch between the WGSL specification and the native language guarantees means that the bounds-checks can be removed even though they must remain to maintain memory safety, creating the WGSL DRSV.

\paragraph{Solution: Constraining Optimizations.}

The optimizations described above must be disallowed so that the bounds-checks remain, i.e., to maintain memory safety. In \myfig\ref{fig:saferace-oob-example-a}, this can be achieved by specifying that all accesses to \texttt{i[0]} be \textit{atomic}. Atomic accesses signal to the compiler that multiple threads might modify \texttt{i[0]}, preventing it from making the assumption that \texttt{i[0]} remains unchanged between $a$ and $b$. In \myfig\ref{fig:saferace-oob-example-b}, the compiler must be prevented from using the value in \texttt{i[0]} to refer to \texttt{x}. While this could be accomplished by again making accesses to \texttt{i[0]} atomic, extending this principle to full programs would require making \textit{every} memory access atomic, causing unacceptable performance decreases for many applications. Instead, this chapter proposes two properties that ensure memory safety violations do not occur due to data races while making fewer accesses atomic. These properties comprise the \saferace Memory Safety Guarantee (\smsg).

The first property of \smsg specifies that slices of programs that contribute to memory indexing must be race-free and can be applied in \myfig\ref{fig:saferace-oob-example-a} by making accesses to \texttt{i[0]} atomic. In more complex code, multiple accesses may contribute to the calculation of the index into \texttt{m}. In these cases, it is not necessary to do the complete index calculation atomically, i.e., by loading all memory and performing arithmetic indivisibly. Rather, it is sufficient to ensure that each sub-access that contributes to the calculation is done atomically, using a backward slicing algorithm that considers all control, index, and data dependencies of an access. \mysec\ref{sec:saferace-smsg} formally shows how this avoids data races on the full index calculation and ensures memory safety.

The second property of \smsg specifies that data races cannot affect the behavior of race-free program slices, preventing the data race on \texttt{i[0]} from affecting the index into \texttt{m} in \myfig\ref{fig:saferace-oob-example-b}. The first property can be implemented as a compiler pass, while the second property must be adopted by intermediate representations targeted by WGSL. \mysec\ref{sec:saferace-smsg} formally defines both properties, while \mysec\ref{sec:saferace-smsg-prop-1} and \mysec\ref{sec:saferace-smsg-prop-2} evaluate the performance and practical impacts of adopting them.

\subsection{\saferace: Threat Assessments and Specification Proposals}

The threat assessments and specification proposals that make up \saferace are described below.

\paragraph{Threat Assessment: Bottom-Up (\mysec\ref{sec:saferace-assess-bu})}
First, the bottom-up threat assessment evaluates the potential for programs with simple memory safety violations, e.g., out-of-bounds accesses without bounds-checks, to cause vulnerabilities. The assessment covers two native GPU frameworks: Vulkan, which runs on Linux and Android operating systems, and Metal, which runs on macOS and iOS. Testing across 11 GPUs from 6 vendors reveals two new memory isolation vulnerabilities, one due to uninitialized memory on AMD GPUs and the other due to out-of-bounds accesses on Apple GPUs. The AMD vulnerability was assigned CVE-2024-36353~\cite{CVE-2024-36353}.

\paragraph{Threat Assessment: Top-Down (\mysec\ref{sec:saferace-assess-td})}
\wgslmemsmith, a fuzzer based on WGSLSmith~\cite{lecoeur2023reconditioning}, is developed to test WGSL implementations for the \drsv, i.e., the data race specification vulnerability described above. \wgslmemsmith generates random programs that contain potential memory safety violations due to combinations of data races and out-of-bounds accesses in order to determine whether WGSL's \drsv can be triggered by existing compilers. Testing \wgslmemsmith on 21 unique compilation stacks, i.e., combinations of browser, operating system, and GPU vendor, did not trigger the specification vulnerability.

Despite this negative result, the vulnerability remains a latent risk, motivating the development of the specification proposals introduced below. Additionally, the fuzzing campaign finds a simpler memory isolation vulnerability due to missing bounds-checks in a pre-release version of Firefox; this vulnerability has been confirmed and fixed by Mozilla developers.

\paragraph{\saferace Memory Safety Guarantee (\mysec\ref{sec:saferace-smsg})}
To fix the WGSL \drsv, this chapter introduces the \saferace Memory Safety Guarantee (\smsg), a contract between a higher-level memory-safe language and lower-level unsafe languages. \smsg consists of two properties: (1)~program slices that contribute to memory indexing, e.g., events $a$, $b$, and $c$ in \myfig\ref{fig:saferace-oob-example-a}, must be race-free, and (2)~data races cannot affect the behavior of race-free slices of programs.

Property 1 can be enforced by an initial pass in WGSL compilers that ensures a program slice is race-free by translating some non-atomic memory accesses into atomic accesses. Property 1 is implemented in Tint, the compiler for WGSL programs included in Chromium-based browsers. Some data types in WGSL, e.g., floats, cannot currently be accessed atomically, so the implementation cannot apply to all possible WGSL programs, e.g., a program with a conditional on floating-point values to determine the index into a memory region. However, the experimental results (\mysec\ref{sec:saferace-smsg-prop-1}) show that data types that cannot be accessed atomically are rarely affected by Property 1 in the tested benchmarks. The evaluation also shows that Property 1 has no measurable performance impact on many existing WebGPU applications, including AI inference and general compute applications, with observed overheads of 3\% and 10\% in two compute kernels. Property 2 ensures that data races do not affect race-free slices of programs and must be incorporated into the frameworks targeted by WGSL, namely Vulkan, Metal, and DirectX. \wgslmemsmith is augmented with the ability to perform metamorphic fuzzing between programs with and without data races and finds violations of Property 2 on only one, likely buggy, machine, providing evidence that these frameworks could adopt this constraint relatively straightforwardly (\mysec\ref{sec:saferace-smsg-prop-2}).

\paragraph{Applications Beyond WebGPU (\mysec\ref{sec:saferace-broader-impacts})}

While the main body of this work focuses on WGSL, the ideas expressed in the threat assessments and specification proposals apply to other parallel programming languages. An increasing number of modern languages are being developed that strive to maintain memory safety through combinations of features like bounds-checked accesses and ownership of data. At the same time, there is a proliferation of tools that can transpile one language to another, and new processors and accelerators with diverse architectures are being developed. This increases the likelihood that layered compilation stacks will become more prevalent, where safety properties must be evaluated across source and target languages.


\subsection{Contributions}

In summary, the contributions of this chapter are:

\begin{itemize}
    \item A bottom-up threat assessment that shows memory safety violations can lead to memory isolation failures in tested GPU frameworks, resulting in the discovery of two new GPU isolation vulnerabilities (\mysec\ref{sec:saferace-assess-bu}).
    \item A top-down threat assessment using \wgslmemsmith to evaluate whether the WGSL \drsv is currently exploitable. The testing did not trigger the \drsv, but revealed a memory isolation vulnerability in a pre-release version of Firefox (\mysec\ref{sec:saferace-assess-td}).
    \item The \saferace Memory Safety Guarantee (\smsg), two properties that, when combined, provide memory safety in the presence of data races in WGSL programs (\mysec\ref{sec:saferace-smsg}).
    \item An implementation of \smsg Property 1 as a WGSL compiler pass and an evaluation of its performance impact on a set of WebGPU applications (\mysec\ref{sec:saferace-smsg-prop-1}).
    \item A metamorphic fuzzing approach implemented in \wgslmemsmith that provides evidence that \smsg Property 2 is supported by the tested systems (\mysec\ref{sec:saferace-smsg-prop-2}).
\end{itemize}

\section{Threat Assessment: Bottom-Up \label{sec:saferace-assess-bu}}

The bottom-up threat assessment motivates memory safety in WGSL by exploring whether memory safety violations lead to memory isolation vulnerabilities in WGSL target languages. As GPUs can be used by multiple concurrent processes on the same machine, any memory isolation failure that leaks data between processes is cause for concern.

The assessment covers two frameworks and three operating systems---Vulkan on Linux and Android and Metal on macOS---as well as three address spaces exposed by WebGPU: device, workgroup, and private (i.e., register) memory. Since some browsers run WGSL programs from different tabs in the same process~\cite{chromium_webgpu_technical_report}, memory isolation is tested both when GPU programs run in different processes and in the same process. The assessment runs two programs concurrently---a victim and an attacker---and tests for memory isolation failures due to uninitialized memory and out-of-bounds accesses.

The victim allocates a block of memory whose size depends on the type of memory being tested. The victim then launches many GPU threads, with each thread assigned a set of memory locations within the block. Each thread alternately writes a canary value and a secret value to even and odd memory locations, respectively. Writing a canary avoids false positives in which the attacker spuriously reads the secret value. The attacker then tests whether it can read consecutive out-of-bounds memory values. The victim's memory is continually reallocated, either automatically in the case of private and workgroup memory or manually in the case of device memory. The victim runs indefinitely until manually stopped.

Like the victim, the attacker allocates a block of memory and assigns each thread a set of memory locations. However, the locations assigned to each attacker thread may be either inside or outside the bounds of the allocated block. The attacker threads iterate over their assigned memory locations and search for the canary. Finding the canary in-bounds indicates that the allocated memory was previously used by the victim and was not properly cleared, while finding it out-of-bounds indicates that the attacker can read data outside its allocated block. If the canary is found, the attacker records the value of the next memory location, and the results are analyzed to determine whether the secret value was found.

In shader languages targeted by WebGPU, memory references generally cannot be manipulated using integer arithmetic like pointers in C-like languages. For example, a shader cannot take a pointer, add an offset to it to create a new pointer, and then dereference that new address. This restricts the ability to write certain algorithms, e.g., arbitrary tree structures. However, Vulkan has introduced a new type of memory, \textit{physical storage buffers}, which allows applications to pass a virtual 64-bit address directly to a shader that can dereference and manipulate it similarly to a pointer. The memory safety implications of physical storage buffers are not well understood, so the assessment covers both device memory using traditional buffer bindings and physical storage buffers in Vulkan.

\subsection{Evaluation \label{sec:saferace-assess-bu-eval}}

\mytablong\ref{tab:saferace-study-results} shows the results of the threat assessment. Overall, the assessment tests 11 GPUs from 6 vendors using 2 frameworks and 3 operating systems. Every tested non-Intel GPU shows a memory isolation failure. Results from private memory are not shown because no failures were observed in this region.

\subsubsection{Same-Process Mode\label{sec:saferace-sp-config}}

This mode follows the setup in many browsers, e.g., Chrome, where GPU programs are run in the same process. Failures observed in this mode are not considered vulnerabilities by vendors, as memory isolation is not guaranteed between applications in the same process. However, in the browser these failures would allow an attacker to read data from another tab if bounds-checks were optimized away, highlighting the danger posed by WGSL's specification vulnerability.

Both the AMD Ryzen 7 5700G, an integrated GPU, and the Arm Mali-G710, a mobile GPU, exhibit uninitialized device-memory isolation failures, while the Apple M2 and M3 GPUs exhibit out-of-bounds access isolation failures. Every tested device that supports physical storage buffers also exhibits failures. While WebGPU does not support physical storage buffers, these results underscore the need for caution when adopting new GPU features. In workgroup memory, the AMD Radeon RX 7900 XT, NVIDIA GeForce RTX 4070, Qualcomm Adreno 610, and Apple M2 exhibit failures. Memory isolation failures due to uninitialized workgroup memory were reported by prior work~\cite{sorensen2024leftoverlocals}, which ran tests in a different-process mode. The results show that some GPUs susceptible to this attack in the different-process mode are also susceptible in the same-process mode. The NVIDIA GPU is the only device that exhibits failures due to out-of-bounds accesses in workgroup memory.

\subsubsection{Different-Process Mode\label{sec:saferace-df-config}}

The assessment also tests the memory isolation of GPU frameworks in a different-process mode, as an attacker running code from the browser might attempt to steal data from other applications running on the same machine. In device memory, failures are observed on two GPUs: the AMD Ryzen 7 5700G and the Apple M2. These failures can be exploited by a malicious process on the same machine to leak data from victim applications or construct covert channels. Each vulnerability is described below. Additionally, uninitialized workgroup-memory isolation failures are observed on the AMD Radeon RX 7900 XT, Qualcomm Adreno 740, and Apple M2, replicating results from previous work~\cite{sorensen2024leftoverlocals}.

\paragraph{AMD Uninitialized Memory Vulnerability}

This vulnerability allows an attacker using AMD's open-source Mesa drivers to read uninitialized memory that can contain data written by another process. Testing confirmed that multiple AMD integrated GPUs are affected, including the Ryzen 7 5700G in this study and a Ryzen 5 5600H. This vulnerability was confirmed by AMD and assigned a CVE~\cite{CVE-2024-36353}, and a patch for the AMD drivers was released.

\paragraph{Apple M2 Out-of-Bounds Access Vulnerability}

This vulnerability allows an attacker to access data from another process by performing out-of-bounds accesses. The extent of this vulnerability is limited---only one 32-bit value can be reliably leaked before driver or hardware protections appear to take effect, perhaps in response to a previous vulnerability that allowed arbitrary access to out-of-bounds memory~\cite{nist_cve_2022_32947}. Nevertheless, the remaining vulnerability is sufficient to construct a reliable, albeit slow, covert channel between two processes. This vulnerability, which affects Apple M1 and M2 GPUs, was reported to and confirmed by Apple.

\section{Threat Assessment: Top-Down \label{sec:saferace-assess-td}}

The top-down threat assessment aims to determine whether the WGSL data race specification vulnerability (\drsv) is exploitable today. This assessment consists of a large-scale fuzzing campaign with \wgslmemsmith, an extension of the WGSLSmith fuzzer~\cite{lecoeur2023reconditioning}. The design of \wgslmemsmith is informed by an understanding of how compilers might optimize away bounds-checks like the one in \myfig\ref{fig:saferace-oob-example}. Therefore, the example from \myfig\ref{fig:saferace-oob-example} is first extended to examine how valid compiler passes in real implementations might introduce memory safety violations. The subsequent discussion shows how \wgslmemsmith generates \textit{patterns} that could trigger the specification vulnerability.

\subsection{Extended Example \label{sec:saferace-extended-example}}

\mylistinglong\ref{list:saferace-c-code} shows C code that implements the same out-of-bounds protections as the GPU example in \myfig\ref{fig:saferace-oob-example}. C is used because the LLVM compiler project enables straightforward exploration of optimization passes. In contrast, optimizing compilers for GPUs are either closed source or do not provide easily configurable tools that allow fine-grained control of optimizations. Although intermediate representations built specifically for GPUs differ from LLVM, the concepts necessary to understand the specification vulnerability in WGSL can be explored using LLVM.

In this example, assume that the length of the \texttt{data} array is 100. The code in red, i.e., the \textcolor{red}{\texttt{min}} check, would be added by a compiler for a memory-safe language. The rest of the code would be written by an application developer (in the memory-safe language). In a single-threaded context, there is no possibility of an out-of-bounds access in this code, but in code running on a parallel processor, a separate thread might create a data race on \texttt{*idx}, writing a value greater than 99.


The LLVM code in \mylisting{\ref{list:saferace-llvm-code}} shows the results of emitting LLVM from the C code using \texttt{clang} and running the following LLVM optimization passes using \texttt{opt}: \texttt{inline}, \texttt{mem2reg}, \texttt{early-cse}, \texttt{instcombine}, and \texttt{simplifycfg}. In the resulting code, a register \textcolor{orange}{\texttt{\%idx\_0}} is allocated that holds the results of the conditional check on the value of \texttt{ptr \%data} (\mylisting\ref{list:saferace-llvm-code} lines 1--3), which corresponds to the conditional on \texttt{data} in \mylisting\ref{list:saferace-c-code} lines 4--7. The value of \textcolor{orange}{\texttt{\%idx\_0}} is stored to \texttt{ptr \%idx} (\mylisting\ref{list:saferace-llvm-code} line 4), which corresponds to the two stores to \texttt{idx} in \mylisting\ref{list:saferace-c-code} lines 5 and 7. Instead of reloading the value of \texttt{ptr \%idx}, the value of \textcolor{orange}{\texttt{\%idx\_0}} is used (extended to an \texttt{i64}) to index into \texttt{ptr \%data} since the compiler assumes the value of \texttt{ptr \%idx} has not changed (\mylisting\ref{list:saferace-llvm-code} lines 5--6). This also allows the compiler to provably remove the \textcolor{red}{\texttt{min}} check in \mylisting\ref{list:saferace-c-code} line 9 when loading from \texttt{ptr \%data\_ptr} (\mylisting\ref{list:saferace-llvm-code} line 7). Note that the conditional in \mylisting\ref{list:saferace-c-code} line 8 is also optimized away in \mylisting\ref{list:saferace-llvm-code}.

In this example, the compiled code is still safe, as the compiler uses \textcolor{orange}{\texttt{\%idx\_0}} instead of loading from \texttt{ptr \%idx}. However, in a more complex program, extra computation could cause register pressure, leading to \textcolor{orange}{\texttt{\%idx\_0}} being spilled to memory before being used to index into \texttt{ptr \%data}. A valid optimization could then rematerialize \textcolor{orange}{\texttt{\%idx\_0}} from \texttt{ptr \%idx}, saving a stack slot\footnote{This same optimization was shown to be problematic when bounding certain data race behaviors in prior work~\cite{dolan2018bounding}.}. Given a data race on \texttt{ptr \%idx}, this value might be larger than the size of \texttt{data}, causing an out-of-bounds access. To the best of current knowledge, this optimization is not implemented in LLVM, although it has been proposed~\cite{llvm_memory_use_markers}. This exploration shows that if the \drsv in WGSL is exploitable, it is likely due to programs that are susceptible to these types of optimizations. The following sections describe the base shaders generated by \wgslmemsmith and show how patterns that might trigger these optimizations are incorporated into the shaders.

\subsection{Shader Generation \label{sec:saferace-base-gen}}

\wgslmemsmith generates shaders in a single pass. It currently supports a subset of WGSL types and expressions, consisting of assignment statements, integer arithmetic, conditionals, and loops. Assignment statements have a variable number of operands in their expressions---by default between 1 and 3. An optional \textit{register pressure} setting increases the number of operands in expressions, up to 20, and prioritizes the use of local variables to increase the number of registers that the compiler might allocate. Overall, shader generation has 16 parameters that can be modified at runtime. These parameters enable comprehensive fuzzing campaigns that randomly generate shaders with different configurations at each iteration.

Along with the base shader, \wgslmemsmith generates racy \textit{patterns} that seek to induce compiler optimizations that could trigger the WGSL \drsv. Patterns are based on templates instantiated during the generation process, with the three types of patterns shown in \myfig\ref{fig:saferace-patterns}. For example, the code in \myfig\ref{fig:saferace-basic-pattern} is an example of a \textbf{Basic} pattern, including potential data races and out-of-bounds accesses. As part of pattern generation, conditionals like the ones in lines 4--6 of \mylisting\ref{list:saferace-c-code} are included, with condition and assignment values randomly chosen. Patterns can be nested in conditionals, loops, or even inside other patterns themselves. Along with basic patterns, \wgslmemsmith generates patterns with other characteristics:

 \begin{itemize}

    \item \textbf{Memory Region.} \wgslmemsmith generates patterns that perform potential out-of-bounds accesses on device, workgroup, and private memory, as different memory regions have different semantics and, as such, might each expose different behavior.

    \item \textbf{Undefined Arithmetic.} Like C++, native GPU languages leave the behavior of signed integer overflow, division/modulo by zero, and division/modulo of the most negative integer by -1 undefined. WGSL compilers include checks to avoid undefined behavior when performing arithmetic. However, a compiler could determine these checks are unnecessary and remove them (e.g., through range analysis). \myfiglong\ref{fig:saferace-arith-pattern} shows an example of integer overflow, where the calculation of the index into \texttt{m} can overflow due to the data race if the arithmetic checks inserted by WGSL compilers are optimized away.

    \item \textbf{Control Flow.} The calculation of an index into memory can be affected by control instructions. If any memory locations that determine the outcome of the conditional participate in a data race, then it is possible that the calculated index may be out-of-bounds. For example, \myfig\ref{fig:saferace-control-pattern} shows an example where a data race could cause the variable \texttt{x} to be assigned to 4, even though the compiler determines that the conditional will not execute.

\end{itemize}

\wgslmemsmith generates programs that run thousands of threads in parallel, taking inspiration from the parallel testing strategy from \mychap\ref{chap:mc-mutants}, with all threads executing patterns and racing with other threads on memory locations used to calculate indexes. Due to nondeterminism in multithreaded execution, each generated shader runs for three iterations to increase the chance of observing vulnerabilities.

\subsection{Evaluation \label{sec:saferace-assess-td-eval}}

We developed a website that runs shaders generated by \wgslmemsmith through WebGPU's JavaScript API and used it to test 21 unique compilation stacks. Overall, \wgslmemsmith ran for over 81 hours and executed tens of millions of shader invocations, with \myfig\ref{fig:saferace-fuzzing-time} breaking down the testing time across different compilation stacks. Fuzzing was performed on both the Chrome and Edge web browsers, but their results are combined because both are built on the open-source Chromium browser and rely on the same underlying WebGPU implementation.

\subsubsection{Testing for Vulnerabilities\label{sec:saferace-assess-td-eval-vuln}}
The fuzzing campaign produced a negative result; data races did not cause memory isolation vulnerabilities due to bounds-checks being optimized away. While this is not a formal proof of the absence of such optimizations, it provides evidence that current compilers might not implement the types of optimizations that could cause the WGSL specification vulnerability. Despite the lack of observable security concerns, the specification vulnerability remains a latent risk to WGSL's memory safety, as future compilers may implement these optimizations.

While testing a pre-release version of Firefox on a Windows laptop with a Qualcomm GPU, non-zero values were observed when reading from out-of-bounds indices in workgroup memory. Investigation showed that the Firefox WGSL compiler was not correctly adding bounds-checks for workgroup and private memory. This vulnerability was privately confirmed by Mozilla and fixed in its implementation of WebGPU~\cite{wgpu-oob}. Because WebGPU was not available in Firefox's release browser at the time of discovery, this vulnerability did not affect most users.

\subsubsection{Mutation Testing} To provide more confidence in the testing approach, mutation testing~\cite{demillo1978hints} is used to analyze the ability of \wgslmemsmith to observe memory isolation failures in the absence of bounds-checks. Bounds-checks could be missing due to compiler bugs, e.g., the Firefox vulnerability described in \mysec\ref{sec:saferace-assess-td-eval-vuln}, or unsound optimizations in the presence of data races, e.g., the examples in \myfig\ref{fig:saferace-oob-example} and \mylisting\ref{list:saferace-c-code}. A flag in the Chromium browser is used to disable bounds-checks on two GPUs, an Apple M2 and M3. As shown in \mysec\ref{sec:saferace-assess-bu-eval}, programs written in Apple's native Metal shading language with memory safety violations revealed memory isolation failures on these GPUs. Therefore, the fuzzing campaign is expected to detect these failures when bounds-checks are disabled. For each mutation-testing trial, \wgslmemsmith is configured to produce one of the three types of patterns from \myfig\ref{fig:saferace-patterns}, i.e., \textbf{Basic}, \textbf{UndefArith}, and \textbf{ControlFlow}.

Each pattern type is then combined with each memory region---private, workgroup, or device---and shaders are generated and run for five minutes on each GPU. The results in \mytab\ref{tab:saferace-mut-testing} show the percentage of generated shaders that led to memory isolation failures on each GPU. Every combination except control-flow dependence with workgroup memory on the M2 showed isolation failures, validating that \wgslmemsmith can observe failures in most tested configurations when bounds-checks are removed.

\subsubsection{Register Pressure}
The final evaluation analyzes whether shaders generated by \wgslmemsmith cause register pressure on various GPUs, which \mysec\ref{sec:saferace-extended-example} describes as a potential cause of dangerous optimizations. The evaluation uses the Vulkan \textit{pipeline executable properties} extension, which allows developers to analyze shader statistics. First, \wgslmemsmith randomly generates a shader using the same limits as the fuzzing campaign. Next, the Tint WGSL compiler generates a SPIR-V shader, and a short program accepts the shader as input, compiles it, and returns shader statistics. The resulting statistics for three GPUs are reported below.

\paragraph{Intel Arc A770 Graphics} Vulkan reports three relevant statistics on this GPU: maximum live registers, spill count, and fill count. Spill count and fill count appear to measure the number of register-spill and reload instructions that the compiler adds to the shader. For the tested shader, the maximum live-register value is 331, the spill count is 683, and the fill count is 1952, showing that register pressure is occurring.

\paragraph{AMD Radeon RX 7900 XT} Vulkan reports several relevant statistics on this GPU: compiler VGPRs/SGPRs, used VGPRs/SGPRs, and scratch-memory usage. VGPR and SGPR stand for vector and scalar general-purpose register, respectively. VGPRs are used when threads in a SIMD unit operate on different data, while SGPRs are used when they operate on the same data. For the tested shader, the compiler reports 256 available VGPRs and 106 available SGPRs and uses 256 VGPRs and 37 SGPRs. The current implementation of \wgslmemsmith focuses on maximizing registers per thread, which explains why SGPRs are not maximized; future versions of \wgslmemsmith might address this deficiency. Scratch-memory usage is 0 when VGPRs are not maximized and increases when they are; for the tested shader, the value is 496 bytes. Therefore, the tested shader can be inferred to cause register spilling on this GPU.

\paragraph{NVIDIA GeForce RTX 4070} The only statistic reported by Vulkan on this GPU is used register count; for the tested shader, this value is 255. This number matches the reported maximum number of registers per thread on NVIDIA's Ada GPU architectures~\cite{nvidia2024cudaada}, meaning that the shader uses all available registers.

\section{The \saferace Memory Safety Guarantee \label{sec:saferace-smsg}}

The threat assessments presented in this chapter, along with vulnerabilities found in prior work~\cite{dipietro2016cudaleaks,lee2014stealing,sorensen2024leftoverlocals,guo2024usenix}, show that GPU memory isolation vulnerabilities continue to be a serious concern. Although the specification vulnerability in WGSL does not currently seem exploitable, the optimizations that would make it so are valid and could legitimately be implemented at any time by GPU compilers. Therefore, this chapter introduces the \saferace Memory Safety Guarantee (\smsg), which addresses the WGSL \drsv by ensuring that bounds-checks are robust in the presence of data races.

Consider a WGSL program $P_u$, which may contain unsafe behaviors. When $P_u$ is compiled to lower-level languages, it is transformed into $P_s$ by inserting checks to guard against unsafe behavior, e.g., \textcolor{red}{\texttt{min}} in \myfig\ref{fig:saferace-oob-example}. However, as described in \mysec\ref{sec:saferace-spec-vuln} and \mysec\ref{sec:saferace-assess-td-eval-vuln}, WGSL compilers include no special protections against data races and may in fact introduce data races. Compiler transformations in the lower-level language can transform $P_s$ into a new program $P_s'$. Under current specifications, if either $P_u$ or $P_s$ contain data races, $P_s'$ may contain memory safety violations, e.g., \myfig\ref{fig:saferace-oob-example} with the \textcolor{red}{\texttt{min}} check removed. To address this, \smsg consists of two properties that, when combined, ensure that $P_u$ is safe from the \drsv. This in turn ensures that in the absence of other undefined behavior (or compiler bugs), valid transformations from $P_s$ to $P_s'$ are also safe.

\begin{itemize}
    \item \textbf{Property 1: Race-Free Indexing Slices:} Program slices that contribute to memory indexing must be race-free.
    \item \textbf{Property 2: Data Race Non-Interference:} In the absence of other undefined behavior, data races cannot affect the behaviors of race-free slices of programs.
\end{itemize}

The remainder of this section formally describes a logic for constructing program slices that contribute to memory indexing and demonstrates how to apply this logic through several examples.

\subsection{A Memory Logic for WGSL \label{sec:saferace-sr-reasoning}}

\mytablong\ref{tab:saferace-sig-reasoning} shows the constructs used to reason about memory safety in WGSL and the languages it targets. WGSL provides a more restricted programming model than other low-level languages, e.g., C++, enabling the construction of the memory logic presented here. A program $P$ operates on data that is either constant, e.g., literals and program constants, or in disjoint memory regions $R$\footnote{Writable memory bound to a shader cannot alias in WebGPU~\cite[\mysec 14.1]{webgpu}.}. A memory region's name is denoted using a subscript, e.g., $R_i$ is the memory region referred to by name $i$. Memory regions can either be private variables/arrays ($R^p_i$), or shared workgroup/device memory ($R^w_i/R^d_i$). Shared memory regions can either be non-atomic ($R^{\{w|d\}na}_i$) or atomic ($R^{\{w|d\}a}_i$).

A memory region is a fixed-size set of 8-bit memory locations. The type of all memory locations in a region is the same, with casting between atomic and non-atomic references to the region prohibited. The set of static memory accesses within a program, i.e., those present in the source code, is represented by $A$. Read and write accesses are represented by $A^r/A^w$, respectively. Declarations of input buffers to a program are represented as a special \textit{initialization} access type $A^i$. Each memory access is given a unique identifier $x$, denoted using a subscript on $A$. Each access $A_x$ has several fields, namely: $A_x.R_i$ denotes the memory region accessed; $A_x.I$ are the index dependencies, i.e., the set of accesses that potentially contribute to the calculation of the index into $A_x$; $A_x.C$ are the accesses on which $A_x$ is control-dependent; and $A_x.D$ are the accesses on which $A_x$ is potentially data-dependent, which for a read access $A^r_x$ are the set of accesses it may read from, and for a write access $A^w_x$ are the set of accesses that determine the value written.

Because the approach is based on static analysis, an access $A_x$ is considered \textit{potentially} dependent on an access $A_y$ if there exists an assignment of values, e.g., due to data races or conflicting atomic accesses, to other accesses in the program that causes $A_x$ to be dependent on $A_y$. Intuitively, this means conditionals based on shared-memory read accesses may resolve to any branch, and a write access with an index dependency on a shared-memory read access may write to any location in a memory region $R_i$. This ensures that any calculation of $A_x$'s dependencies is sound but potentially over-approximated.

The transitive closure of the union of $A_x.I$, $A_x.C$, and $A_x.D$ is called the access \textit{slice}, or $A_x.S$. A slice represents all memory accesses that may affect $A_x$, including its reachability, the memory locations it accesses, the value stored if $A_x \in A^w$, and the value it returns if $A_x \in A^r$. $A_x.S$, along with its constituent sets, can be obtained via static program slicing based on a data-flow and control-flow analysis~\cite{weiser1984slicing}.

The definition of an access enables precise reasoning about programs with \smsg. Property 1 can be enforced by a compiler in a safe high-level language via the rule \textsc{RaceFreeSlice} in \mytab\ref{tab:saferace-sig-reasoning}. This rule specifies that given an access $A_x$, for all accesses $A_y$ in $(A_x.I \cup A_x.C)$, if $A_y.R_i$ is a shared memory region it must be made atomic. Additionally, for all accesses $A_z$ in $A_y.S$, if $A_z.R_i$ is a shared memory region, it must also be made atomic (note that \texttt{mkAtomic()} is a no-op for private memory regions and that when applying \textsc{RaceFreeSlice} to $A_x$, $A_x.D$ is not included because the data dependencies of $A_x$ do not determine the memory locations in $R_i$ accessed by $A_x$). Intuitively, the program slice for each index and control dependency of an access is now race-free as all contributing memory accesses have been made atomic, preventing the optimizations discussed in \mysec\ref{sec:saferace-spec-vuln} and \mysec\ref{sec:saferace-extended-example}.

Property 2, which must be adopted by the memory-safe language \textit{and} the languages in its compilation stack, specifies that if a memory access $A_y$ is not in $A_x.S$, it cannot affect the result of $A_x$ even if it participates in a data race. Therefore, when running a program $P$, the possible behaviors of $A_x$, i.e., \texttt{bhvrs}($A_x$), with $A_y$ included in $P$ must be equivalent to the possible behaviors of $A_x$ when running $P$ with $A_y$ removed.

\subsection{Applying \smsg}

The application of \smsg is first demonstrated on the simple examples in \myfig\ref{fig:saferace-oob-example}, followed by a more complex example that includes control flow and multiple types of dependencies.

\paragraph{Simple Example}
Both programs in \myfig\ref{fig:saferace-oob-example} contain two memory regions assumed to be device-scoped---\texttt{i} ($R^{dna}_i$) and \texttt{m} ($R^{dna}_m$)---and one private memory region---\texttt{x} ($R^p_x$). In \myfig\ref{fig:saferace-oob-example-a}, the access to \texttt{m} is index-dependent on the access to \texttt{i[0]}, so applying \textsc{RaceFreeSlice} means compiling $R^{dna}_i$ to $R^{da}_i$, thereby making all accesses to \texttt{i[0]} atomic. Because accesses are now atomic, the compiler cannot assume that \texttt{i[0]}'s value will not change between accesses and therefore cannot erroneously remove the bounds-check. In \myfig\ref{fig:saferace-oob-example-b}, $R^p_x$ is race-free as it is a private memory region, and applying \textsc{RaceInterference} means that the data race on $R^{dna}_i$ cannot affect it.

\paragraph{Larger Example}
\mylistinglong\ref{list:saferace-smsg-reasoning} shows an example of how to apply this logic to a program $P_u$. There are six memory regions in $P_u$: $R^{dna}_a$, $R^{wna}_b$, $R^{wna}_c$, $R^{dna}_d$, $R^p_x$, and $R^p_r$. All memory events, including the declaration of buffers on lines 1--4, are represented as memory accesses. Only non-empty base fields are shown, with the type of a memory region omitted after its first reference.

Accesses and their dependencies are shown either above or on the line they reference, so the accesses on lines 12--15 all correspond to the code in line 15. Working backward from line 15, $A^r_{13}$ is index-dependent on $A^r_{11}$ and $A^r_{12}$. $A^r_{11}$ is potentially data-dependent on $A^w_6$ and $A^w_8$, while $A^w_8$ is control-dependent on $A^r_7$. Meanwhile, $A^r_{12}$ is data-dependent on $A^w_{10}$, which in turn is data-dependent on $A^r_9$. At this point, every memory region besides $R_d$ and $R_r$ in $P_u$ is contained in accesses in $A^r_{13}.S$.

A compiler for an unsafe language can assume the absence of data races because all memory regions are non-atomic. Under this assumption, all bounds-checks in this example could be provably removed as the index on line 15 evaluates to 5. However, data races may cause these values to change in ways that break memory safety. For example, a race on \texttt{c[0]} or \texttt{b[0]} could cause their values to increase even after the compiler has removed the bounds check, as \myfig\ref{fig:saferace-oob-example} and \mylisting\ref{list:saferace-c-code} showed. Similarly, a race on \texttt{a[0]} could cause the code in the conditional to execute despite the compiler optimizing based on the determination that it is dead code\footnote{Since it is dead code, a compiler might decide to eliminate it, in which case $R_a$ does not need to be made atomic. However, GPUs execute in a single-program multiple-data manner, so dead code in one thread may not be dead in another and a compiler may decide to leave it in place.}.

When compiling from $P_u \rightarrow P_s$, applying Property 1 via \textsc{RaceFreeSlice} means compiling memory regions as follows: $R^{dna}_a \rightarrow R^{da}_a$, $R^{wna}_b \rightarrow R^{wa}_b$, and $R^{wna}_c \rightarrow R^{wa}_c$. Legal transformations $P_s \rightarrow P_s'$ cannot assume the value of atomic memory locations, prohibiting optimizations that both remove the bounds check and rematerialize from the shared memory regions in accesses in $A^r_{13}.S$. The other memory region in accesses in $A^r_{13}.S$, $R^p_x$, is not made atomic; \textsc{RaceInterference} ensures that its value cannot be affected by data races elsewhere in the program.

\subsection{\smsg Soundness}

The following argument shows that the rules for applying \smsg ensure that data races cannot lead to memory safety violations. Consider a program $P_u$ that contains a memory access $A_x$ that may attempt to read out-of-bounds memory. In an initial compilation from $P_u \rightarrow P_s$, e.g., WGSL to a lower-level language, $A_x$ is compiled to $A'_x$, which is guaranteed not to read out-of-bounds due to checks added by the compiler. Now, assume a valid transformation $T$ in the lower-level language compiles $A'_x \xrightarrow{T} A''_x$, and that $A''_x$ no longer contains bounds-checks and leads to a memory safety violation.

If Property 1 is applied correctly, there are no data races in $B$, the set of memory accesses in $A'_x.I$ and $A'_x.C$ and in their slices. Therefore, any analysis the compiler performs to remove the bounds-check in $A''_x$ while considering the values of memory accesses in $B$ is safe. The only other sources of data races in $P_s$ are accesses not in $B$, which cannot affect the behaviors of accesses in $B$ according to Property 2. As $B$ is exactly the set of accesses that contribute to the calculation of $A'_x$'s (and therefore $A''_x$'s) index, $A'_x$ and $A''_x$ are safe. This means that if $T$ is a valid transformation, it must have introduced the memory safety violation due to behaviors other than data races, validating the claim that $P_s$ is safe in the presence of data races.

\paragraph{Complete Memory Safety in WebGPU}

Enforcing \smsg ensures that data races no longer lead to globally undefined behavior or memory safety violations due to out-of-bounds accesses. However, to provide complete memory safety, WebGPU needs to guarantee that no possible WGSL program can lead to undefined behavior in its lower-level language targets. The current WGSL specification avoids undefined behavior by classifying its common sources, including data races, as dynamic errors, but the preceding analysis shows that this is not necessarily sufficient. Therefore, the analysis considers the other sources of dynamic errors mentioned in the specification, which are: aliased writable memory bound to a shader~\cite[\mysec 7.3]{wgslspec}, mismatched memory layouts~\cite[\mysec 13.4]{wgslspec}, unbounded loops~\cite[\mysec 9.4.3]{wgslspec}, and an erroneous graphics-specific vertex value~\cite[\mysec 12.3]{wgslspec}.

Non-aliasing of memory is enforced by the WebGPU host API~\cite[\mysec 14.1]{webgpu}, and while mismatched memory layouts between the host and shaders can affect program correctness, the values in memory do not change the analysis used to apply \smsg. Graphics pipelines are not considered as they are commonly structured around per-vertex and per-fragment inputs and outputs~\cite[\mysec 13.1, 13.3.1--13.3.2]{wgslspec}. However, \smsg techniques could apply to graphics shaders that use shared writable resources, which would be a natural extension to this work. This leaves unbounded loops as the last source of dynamic errors. Unlike in languages like C++, where unbounded loops with side effects are useful for many tasks, GPU languages are generally not designed for long-running programs. In fact, most GPU languages currently lack a well-specified forward progress model~\cite{sorensen2021progress}, so even unbounded loops with side effects often lead to crashes or other issues. The lack of support for unbounded loops has implications for memory safety, and open-source WGSL compilers insert code in response to the fact that the Metal compiler might optimize away bounds-checks because it considers infinite loops without side effects undefined behavior~\cite{wgpuIssue4972}. While this approach appears successful so far, the lack of specificity around the behavior of unbounded loops means that the \smsg guarantees currently apply only to source programs that are free of behaviors like unbounded loops.

\subsection{Other Memory Safety Approaches}

There are several other approaches that could be used to avoid the removal of bounds-checks and help maintain memory safety in WGSL. The first, and simplest, is to enforce that all memory accesses are made atomic when compiling from WGSL to lower-level GPU languages, which would remove data races from all programs. However, even though the \texttt{relaxed} memory order semantics for atomics used by \smsg are less costly than more heavyweight memory orders like \texttt{acquire}/\texttt{release} and sequentially consistent, the results in \mysec\ref{sec:saferace-smsg-prop-1-eval} show that in cases where some accesses must be made relaxed atomic operations, there is already a performance decrease. Making every access atomic likely would cause unacceptable performance for many applications.

Another option is LLVM's \texttt{unordered} atomic ordering~\cite[Atomic Memory Ordering Constraints]{LLVM_LangRef} for all accesses, which provides weaker semantics than \texttt{relaxed} but prevents rematerializing loads and is used by LLVM to match Java's memory model. However, the languages targeted by WGSL, e.g., SPIR-V, Metal, and HLSL, do not all explicitly support unordered atomics yet. To implement this solution, the changes in GPU compiler implementations and the performance of using unordered atomics would need to be carefully studied and tested. Additionally, \smsg can be integrated with unordered atomics, as Property 1 can translate access dependencies to unordered atomics instead of relaxed atomics, while Property 2 is weaker than unordered atomic guarantees.

A different style of approach is to integrate some form of ``unoptimizable checks'' into WGSL and the languages it targets. For example, LLVM has a concept of \textit{linkage types}~\cite[Linkage Types]{LLVM_LangRef}, including types like \textit{weak} that prevent a compiler from removing unreferenced global variables and functions. If the \texttt{min} function is declared using this attribute, the compiler would therefore not be able to remove it. While this approach is promising, unoptimizable checks would require a large amount of coordination and implementation work across WGSL and the languages it targets. Also, to the best of current knowledge, the guarantees offered by existing unoptimizable checks like linkage types are not formally defined in relation to existing sources of undefined behavior such as data races. On the other hand, \smsg is a novel solution to memory safety in the presence of data races using existing language constructs that are preserved across all target languages, i.e., relaxed atomic accesses.

\section{Implementing \smsg Property 1 \label{sec:saferace-smsg-prop-1}}

Property 1 was implemented as a pass in Tint, the WGSL compiler included in the Dawn WebGPU implementation used by Chromium. While Tint is not an optimizing compiler, it does include significant infrastructure for performing passes on WGSL programs, e.g., inserting bounds-checks and ensuring arithmetic expressions are well defined. Tint converts WGSL programs into an intermediate representation (IR) in static single-assignment (SSA) form and includes utilities for traversing and modifying the IR.

The pass consists of two phases: a backward slicing phase and a forward conversion phase. In the slicing phase, the \textsc{RaceFreeSlice} rule is applied to compute the sets of accesses that contribute to an access $A_x$. In the conversion phase, non-atomic memory regions are converted to atomic regions, and every use of these regions is updated to atomically load from or store to them. The implementation accounts for nested types, e.g., arrays and structs, handles control flow, e.g., loops and branches, and performs complete interprocedural analysis. Implementing the pass revealed several constructs that either provide opportunities for optimization or expose limitations in WGSL. These constructs are detailed below because they provide interesting avenues for future WGSL development.

\paragraph{Read-Only Regions} All WGSL memory regions have an \textit{access mode}, either \texttt{read}, \texttt{read\_write}, or \texttt{write}. During the calculation of Property 1, read-only regions are not converted to atomic because, by definition, reads cannot race with each other in the absence of a write. However, in some cases a region marked as \texttt{read\_write} is only read during execution of the shader. Although this optimization is not implemented, regions that are only read from, even if marked \texttt{read\_write}, do not need to be accessed atomically.

\paragraph{Limitation on Atomic Types} WGSL currently allows only two atomic types: unsigned and signed 32-bit values. This limitation has implications for both the completeness and performance of the implementation of Property 1. For example, one element of a \texttt{vec2} may be included in the index dependency of an access, but because a \texttt{vec2} cannot be made atomic, it cannot be converted to ensure that it will not participate in a data race. Similarly, if a floating-point value is part of the control dependency of an access, the region from which the floating-point value is loaded cannot be converted. Therefore, it is currently not possible to implement Property 1 completely in WGSL. When the implementation encounters a data type that cannot be accessed atomically in WGSL, it is flagged so that users of the compiler pass can be made aware of the issue. For example, safety-critical applications may decide to discard flagged shaders and perform the computation on the CPU instead. In the future, WGSL could add support for atomic floats and other data types, following the example set by C++, which implements an atomic type template that can be applied as long as certain \textit{named requirements} are satisfied~\cite{cppreferenceAtomic}.

The limitations on atomic types also impact the performance of shaders translated by the pass. In WGSL, atomic types are not \textit{constructible}, i.e., able to be loaded or stored directly, passed into functions, or created outside of the declarations of buffer types. If a struct or sized array that contains a type converted by the pass is loaded into a private variable, a function loads each field of the struct or loops over the array and loads each element. It is not possible to directly load a runtime-sized array, creating an upper bound on the number of accesses that must be made atomic. WGSL's ability to support atomic types is dependent on their implementation in native GPU languages. These languages are slowly adding support for more atomic types, so future enhancements to WGSL can increase the coverage of \smsg Property 1. For full completeness, however, it may be required to limit the types that can contribute to memory accesses.

\paragraph{Interprocedural Analysis} Every WGSL program is contained in a single file, with a single entry-point function that is dispatched on the GPU by the host. This allows complete interprocedural analysis to be performed on the program, traversing function calls to search for contributing shared memory accesses. Shared memory regions may either be passed into functions as pointer arguments or directly accessed from within the function. The analysis also requires mapping function parameters to the arguments with which they are called, as the return value of a function may have data dependencies on shared memory regions from which the arguments were loaded.

\paragraph{Targeting Different Backends}
As described in \mysec\ref{sec:bg-mem}, both DirectX and Vulkan provide guarantees that operations on device memory will not execute out-of-bounds, while Metal does not. It may therefore seem unnecessary to apply \smsg Property 1 to device-memory accesses in DirectX and Vulkan. However, these guarantees are more analogous to runtime protections in operating systems than to features of the languages themselves. Data races in both HLSL and SPIR-V programs are undefined, meaning that memory safety violations could potentially occur in racy programs. In some sense, these could also be considered specification vulnerabilities of DirectX and Vulkan, motivating future work on the development of these frameworks. Therefore, \smsg Property 1 is applied to all memory operations on all backends.

\subsection{Evaluation \label{sec:saferace-smsg-prop-1-eval}}

While the goal of \smsg is to prevent malicious actors from exploiting memory safety violations in WGSL programs, its guarantees apply to all WGSL programs. This means that even in programs that are not malicious, the \smsg pass analyzes memory accesses and translates dependencies into atomic operations. Therefore, a study was conducted to evaluate the cost of \smsg Property 1 in terms of both compilation time and GPU performance.

\subsubsection{Benchmarks}

The implementation of Property 1 in Tint enables a custom version of Chromium on an Apple M3 to run the analysis on a set of WebGPU applications, detailed in \mytab\ref{tab:saferace-webgpu-benchmarks}. As WebGPU is still in active development and is not yet fully available on all web browsers, this set of applications consists mostly of demonstrations. These applications provide a benchmark suite covering several ways WebGPU can be used by browser applications.

The first five benchmarks in \mytab\ref{tab:saferace-webgpu-benchmarks} are AI applications. \textbf{WebLLM DeepSeek Qwen 7B} and \textbf{WebLLM Llama 3.2 1B} use WebLLM~\cite{ruan2024webllmhighperformanceinbrowserllm} to run LLM inference. Under the hood, WebLLM relies on Apache TVM~\cite{apache_tvm}, specifically TVM's Web runtime, and supports many models. The benchmark includes two models: the Llama-3.2-1B-Instruct generative model and the DeepSeek-R1-Distill-Qwen-7B reasoning model. \textbf{ONNX Runtime (Phi-3)} and \textbf{ONNX Runtime (Whisper)} use ONNX Runtime Web~\cite{onnxruntime_webgpu_tutorial} through transformers.js~\cite{transformersjsDocs}, which allows machine learning models to be deployed and run in browsers using WebGPU. The \textbf{Phi-3} benchmark runs inference using the Phi-3-mini-4k-instruct model, while the \textbf{Whisper} benchmark runs speech recognition with OpenAI's Whisper Large V3 Turbo. Finally, \textbf{MediaPipe Gemma2 2B} uses Google's MediaPipe suite to run LLM inference with the Gemma2 2B model.

The other benchmarks consist of demonstrations of WebGPU's compute abilities. Three, \textbf{Bitonic Sort}, \textbf{Game of Life}, and \textbf{Compute Boids}, are part of the WebGPU Samples maintained by the official WebGPU working group~\cite{WebGPUSamples}, \textbf{FFT Ocean Demo} shows how to use compute shaders to render complex ocean waves as part of the Babylon.js library for 3D web graphics~\cite{FFTOceanDemo}, and \textbf{MSM} is a WebGPU implementation of multi-scalar multiplication submitted to the 2023 ZPrize competition~\cite{ZPrize2025}. This last benchmark relies on sparse-matrix multiplication techniques, an area where complex index dependencies may be more common than in dense computations.

\subsubsection{Compilation Cost}

Unlike a language like CUDA, which is often precompiled to GPU assembly code, WGSL shaders are translated to native GPU languages and vendor-specific machine code at runtime. To mitigate the cost of runtime compilation, WebGPU implementations and native GPU frameworks use layered and nonstandardized compilation processes, often involving asynchronous operations, just-in-time (JIT) compilation, and caching components. This complex structure makes it difficult to measure the timing of the full compilation pipeline. Therefore, rather than reporting the time taken by the pass as a percentage overhead of the full compilation process, \mytab\ref{tab:saferace-webgpu-benchmarks} reports the raw time, in the \textbf{Pass Timing} column, required to run the pass on the shaders in each benchmark.

Compared against only the time taken to run the existing passes in the \texttt{tint} compiler, the addition of the \smsg pass adds a large overhead, up to $9\times$ for the slowest-compiling shader in the \textbf{MSM} benchmark. However, these pre-existing passes are generally much simpler, e.g., performing a sequential scan of a shader and adding bounds-checks to memory accesses, while the \smsg pass requires relatively complex dependency analysis and program traversal. The timing of passes in the \texttt{tint} compiler also does not include the time required to parse WGSL programs into an IR and write them out to native GPU programs, nor the time taken by native-language compilers to generate machine code, meaning that the time taken to run all passes is a small part of the overall compilation and runtime process. For example, on the \textbf{WebLLM DeepSeek Qwen 7B} benchmark, the one-time cost of the \smsg pass summed across all shaders was only 62.47 ms, while each shader ran dozens to hundreds of times for a total of several seconds during a \textit{single} inference response, with typical LLM interactions usually consisting of many inferences.

\subsubsection{GPU Performance Cost}

Atomic operations constrain compiler optimizations and thus can negatively impact performance~\cite{schweizer2015atomiccost,burtchell2024characterizing}. Therefore, it is likely that any atomic operations added to satisfy \smsg Property 1 could lead to reduced application performance. However, as shown in the \textbf{Runtime Overhead} column of \mytab\ref{tab:saferace-webgpu-benchmarks}, \smsg Property 1 did not apply to any shaders in the majority of the tested benchmarks. The four benchmarks where it did apply are discussed below.

While the two WebLLM model benchmarks differed in the number of shaders used by the program, they both compile a shader called \texttt{apply\_penalty\_inplace\_kernel} that requires making four memory regions atomic, affecting 24 accesses in the program. The developers of WebLLM confirmed that these shaders are not yet used during LLM inference. Additionally, the accesses that must be made atomic in this shader are floating-point accesses, and as mentioned in \mysec\ref{sec:saferace-smsg-prop-1}, WGSL does not yet allow atomic access to floating-point data types. Therefore, if WebLLM starts using this shader, the current implementation of Property 1 cannot provide its memory safety guarantee for the shader, at least until WGSL begins to support atomic accesses to more data types.

The \textbf{Bitonic Sort} benchmark does require making one workgroup-buffer memory region atomic, affecting 10 accesses in one shader. To complete one full sort using default parameters, this shader runs 91 times. To obtain precise GPU timing information, Dawn's WebGPU implementation was extended to automatically record GPU timestamps for every compute shader. The full sort was run three times each with the pass enabled and disabled on an Apple M3 GPU, and the average time across the 91 runs of the shader was calculated. On average, each shader run took between 0.04--0.09 ms, for a total of 3.6--8.2 ms. With the pass enabled, the average runtime of the shaders increased by 9.8\%, from 0.064 ms per shader to 0.071 ms. The same analysis was performed on the \textbf{MSM} benchmark; applying \smsg Property 1 to these shaders required making seven accesses atomic. The average shader runtime of this benchmark increased from 0.99 seconds to 1.02 seconds, or 3\%.

To conclude, these results show that only \textasciitilde 2\% of the shaders in the benchmarks are affected by \smsg Property 1. The benchmarks consist of important WebGPU applications in several domains, including LLM inference and general compute algorithms, and are either generated by state-of-the-art tensor compilers, e.g., TVM~\cite{apache_tvm}, or handwritten by expert GPU developers. Many of the shaders used by these applications do not include the complex indirect memory accesses that require transformations to satisfy \smsg Property 1. For example, matrix multiplication is generally written using simple index calculations taken from combinations of thread identifiers and constants, and none of the matrix multiplications in the LLM benchmarks require any transformation. This suggests a promising direction for future work, as safety analyses for GPU programs may be able to take advantage of the regular indexing structure common in tensor compiler outputs to provide stronger guarantees for important machine-learning workloads at relatively low cost. On the other hand, some applications, e.g., bitonic sort, may experience reduced performance due to the safety requirements of \smsg Property 1. The results also show that in this set of benchmarks, data types that cannot be accessed atomically in WGSL are generally not used in index-dependence calculations, with the only exception being the WebLLM shader that may be used in the future. All browser-based languages must make tradeoffs between performance and safety, and these results provide evidence that the performance overhead of adopting \smsg Property 1 may be acceptable for some applications.

\section{Testing \smsg Property 2 \label{sec:saferace-smsg-prop-2}}

While \smsg Property 1 can be implemented by WGSL compilers, Property 2 must be adopted by the languages targeted by WGSL. This may seem like a thorny problem; many languages, including C++ and GPU languages, leave the behavior of data races undefined to avoid having to reason about constraints like the one required by Property 2. However, previous work on data races has shown how to bound the behavior of data races to provide a local data-race-free (Local DRF) property and has proven the soundness of many existing compiler optimizations given this property~\cite{dolan2018bounding}.

The Local DRF property is stronger than \smsg Property 2. It prevents a data race on one memory location $x$ from causing an unrelated memory location $y$ to read a value written to $x$, a guarantee called \textit{bounding data races in space}. It also prevents the effects of data races from leaking across synchronization points, e.g., barriers, a guarantee called \textit{bounding data races in time}. In particular, bounding data races in time requires preserving load-store ordering, which has been shown to have relatively low cost on CPUs. However, the cost of preserving load-store ordering has not been studied on GPUs, where load-store reordering is very commonly observed, as shown in \mychap\ref{chap:gpuharbor}. Still, prior work showed that preserving load-store ordering is not necessary to bound data races in space and proved that many existing compiler transformations are compatible with this property~\cite{dolan2018bounding}, and therefore with \smsg Property 2.

To test whether \smsg Property 2 is maintained by current GPU compilers, the \wgslmemsmith fuzzer was augmented using metamorphic fuzzing. When generating shaders, variables and memory locations are split into \textit{safe} and \textit{unsafe} sets. Safe memory locations are read-only or read/written by only one thread, while unsafe memory locations may be read or written to by any thread. Safe local variables are only used in combination with safe memory locations, while unsafe variables can be combined in expressions with unsafe memory locations. Intuitively, this process creates race-free slices of programs, as well as slices that may contain races. During fuzzing, two shaders are run: one with only race-free slices and the other with both race-free and racy slices.

The race-free shader is run first, and the values in memory at the end of execution are saved. Since this shader contains no data races or sharing of memory between threads, the results are deterministic. The racy shader, which contains the exact statements in the safe shader along with statements that operate on unsafe memory locations and may contain data races, is run next. The values in safe memory locations at the end of the racy shader execution are then compared with the values from the race-free shader execution. If the racy shader results differ, the addition of data races may have caused the differences, violating \smsg Property 2.

\subsection{Evaluation}

Because the same shaders and testing infrastructure are used for both the metamorphic fuzzing and the top-down threat assessment, the evaluation covers the same set of compilation stacks and testing time shown in \myfig\ref{fig:saferace-fuzzing-time}. On most tested compilation stacks, data races did not affect race-free slices of programs. This was true for all compilation stacks using Vulkan and DirectX as the WebGPU backend, as well as Metal backends targeting Apple M-series GPUs. On DirectX backends, data races caused unexpected values in unsafe memory locations, i.e., values that could not be explained by any writes to those memory locations, but this behavior did not affect race-free slices of the programs.

On one machine, an older MacBook with an Intel SoC, the racy shader caused memory in race-free slices of the shader to change. A browser-based reducer built using fuzzing reduction techniques~\cite{zeller2002deltadebugging,regehr2012creduce} determined that the mismatches between the reduced shader outputs were due to an unrelated issue. This issue manifests in the presence of certain specific loop constructs and occurs when running a single-threaded shader containing no data races on the GPU. The issue appears to be a compiler or driver bug and was reported to Apple.

These observations provide evidence that \smsg Property 2 holds in the tested WebGPU implementations, which should ease its adoption by GPU languages. Besides memory safety, \smsg Property 2 also allows more compositional reasoning about program behaviors, enabling other future optimizations and safety properties.

\section{Summary and Broader Impact \label{sec:saferace-broader-impacts}}

The proposals in this chapter are designed to address WGSL's specification vulnerability, but the results can also inform the general development of safe languages that target parallel processors and seek to provide memory safety and reasonable semantics for data races. The work has potential lessons and impacts for three different areas: existing browser-based languages, existing native languages, and the future development of languages that require compositional safety reasoning.

\subsection{Browser-Based Languages}
The other two main languages used in the browser, JavaScript and Wasm, both require bounds-checks for memory safety and constrain the effects of data races, including specifying that data races must be bound in space. %While Local DRF simplifies program reasoning, it also forbids behaviors that have been observed on GPUs, like load buffering~\cite{levine2023gpuharbor}. In fact, our pilot experiments showed that the effects of data races \textit{can} cross synchronization boundaries on GPUs, further hampering the full adoption of Local DRF by GPU languages.
Unlike WGSL, the semantics of these languages can also be fully controlled by an implementation. For example, the V8 JavaScript and Wasm engine controls the optimization phases of compilation and directly generates machine code, avoiding intermediate languages with undefined behavior. In some cases, these engines implement bounds-checks by relying on operating systems or processors to fault when memory is accessed out-of-bounds~\cite{watt2019weakening}, a guarantee that GPU runtimes do not necessarily provide.

Even so, the problems associated with optimizations on bounds-checks and data races must be considered by these languages. JavaScript and Wasm engines may target architectures without built-in memory protections, so they must not implement optimizations that both remove bounds-checks and rematerialize loads. Additionally, as described in \mysec\ref{sec:saferace-base-gen}, dead code may execute due to data races in control dependencies, so optimizations on those blocks must be carefully checked. %Handling the effects of dependencies is in general a complex problem, and has been shown to be the root of out-of-thin-air executions in general-purpose programming languages~\cite{batty2015probconcurrency}.

\subsection{Native Languages}

Some native languages, such as Java and Go, give weak semantics to data races and control their compilation stacks, similar to the browser-based languages. Rust takes a different approach, using its ownership system to avoid races and memory safety violations in Safe Rust. However, Unsafe Rust, along with other unsafe languages like C++, does not provide the same guarantees. A large body of research exists to retrofit unsafe languages with memory protections, including runtime analyses like AddressSanitizer~\cite{serbryany2012asan} and pointer augmentation like Checked C~\cite{ruef2019checkedc}. However, undefined behavior, e.g., due to data races, makes it difficult to reason about these protections' soundness.

In contrast, if a native language were to adopt \smsg Property 2 and apply \smsg Property 1 to indirect accesses via its compiler, it would be able to provide more robust memory safety guarantees. In general, \smsg would be difficult to apply to large, feature-rich C++ codebases spanning many files and using complex language features, as the Property 1 compiler pass needs access to the full program slice to compute the full access dependency graph and cannot easily handle features like pointer aliasing, dynamic allocations, and linkage types. However, in safety-critical settings, e.g., automotive, there is already precedent, e.g., MISRA-C++~\cite{misracpp}, for restricted subsets of languages where Property 1 could more easily apply. For example, a C++ dialect where all pointers are restricted from aliasing, memory locations are not accessed both atomically and non-atomically, and mixed-size accesses are restricted would correspond almost exactly to WGSL, in which case \smsg could apply directly. This is a promising avenue of future research, with the tools and proposals in this chapter providing a framework for formally defining such a language dialect.

\subsection{Compositional Language Design}

WGSL is somewhat unique in that it both needs to enforce memory safety and relies on intermediate languages that are inherently weaker with regard to these protections, motivating the development of \smsg. However, this type of compositional language design is likely to become more prevalent. The post-Dennard scaling landscape has led to the development of numerous types of architectures and associated software stacks, while frameworks like LLVM and MLIR are paving the way for increasingly connected pathways from one language to another.

As shown in this chapter, understanding the interactions between languages and their safety guarantees is complex. The approach blends formal language proposals with extensive empirical evaluation. These results motivate future research that seeks to clearly specify behaviors that have classically been left undefined, such as data races, making it easier to compose language semantics and enable performant and safe applications.

Overall, this chapter shows that GPU programming systems need safety guarantees that survive the full path from high-level source languages to lower-level compiler targets and hardware. The threat assessments demonstrate that memory safety violations in GPU programs can turn into memory isolation vulnerabilities, while the \smsg proposal shows how language and compiler rules can prevent data races from undermining bounds-checks. Together, these results show that GPU specifications are most useful when formal reasoning is combined with empirical validation.
